## Firebase (M1.2 state: real projects provisioned)

The committed per-flavor options (`lib/core/firebase/firebase_options_*.dart`)
hold REAL config for **`hayatiapp-dev`** / **`hayatiapp-prod`** (issue #5,
Session 004): `flutterfire configure` per flavor, Google provider enabled on
both, per-flavor OAuth client ids in `lib/core/firebase/google_sign_in_config.dart`
(iOS `clientId` at runtime, Android `serverClientId`; both iOS
`REVERSED_CLIENT_ID` URL schemes registered in `ios/Runner/Info.plist`).
Config files are identifiers, not secrets ŌĆö committed deliberately
(`docs/architecture.md` ┬¦2). Each project has a `(default)` Firestore
database at `eur3` with `firestore.rules` deployed.

So `flutter run -t lib/main_dev.dart` with no dart-defines talks to the real
`hayatiapp-dev`. Local emulator work stays opt-in (`firebase.json` at the
repo root; the **Firestore emulator needs Java 21+ on PATH**, auth-only runs
do not):

```sh
# terminal 1 ŌĆö repo root
npx firebase-tools emulators:start --only auth,firestore --project demo-hayati

# terminal 2 ŌĆö app/, any device/simulator
flutter run -t lib/main_dev.dart \
  --dart-define=USE_AUTH_EMULATOR=true \
  --dart-define=USE_FIRESTORE_EMULATOR=true
# physical device: add --dart-define=AUTH_EMULATOR_HOST=<your LAN IP>
# (shared by both emulators)

# integration tests (device/simulator only ŌĆö firebase_auth needs platform
# channels, so these never run in the `flutter test` VM)
flutter test integration_test --dart-define=USE_AUTH_EMULATOR=true \
  --dart-define=USE_FIRESTORE_EMULATOR=true -d <device>
```

The suites cover Google, Apple, and phone sign-in plus the profile round-trip.
Apple and Google are faked with unsigned JSON id_tokens (the emulator does not
verify them); phone codes are minted randomly by the emulator and read back from
`/emulator/v1/projects/demo-hayati/verificationCodes` ŌĆö note that path carries
the project the **emulator was booted with**, not the app's `projectId`.

In CI these run in the `integration-emulator` job, which is **main-push-only**
because macOS minutes bill at 10├Ś (see `docs/architecture.md` ┬¦9). To exercise it
on a branch before merging: `gh workflow run ci.yml --ref <branch>`.

The phone suite is **quarantined from that job** (issue #15): `verifyPhoneNumber`
crashes the app natively on the iOS simulator. Google, Apple, and the profile
round-trip do gate it. The phone flow is covered by VM unit tests meanwhile.

## Conventions

- Flavors are Dart entrypoints (`main_dev.dart` / `main_prod.dart`) overriding
  `appConfigProvider`; store-level flavor split arrives with CI/Fastlane (M0.2).
- Brand strings live only in `lib/core/config/` (working title pending
  trademark search ŌĆö `docs/frontend-brandkit.md` ┬¦1).
- Riverpod codegen: generated `*.g.dart` files are committed so a fresh clone
  tests green without a build step; regenerate with
  `dart run build_runner build --delete-conflicting-outputs`.
- l10n: ARB sources in `lib/core/l10n/arb/` (en template + tr + ar) are the
  copy of record; the generated `lib/core/l10n/gen/` is **gitignored** and
  regenerated by `flutter pub get` / `flutter gen-l10n` (NOT by
  `flutter test` ŌĆö run pub get after editing an ARB). Tests assert via
  `test/support/localized_app.dart` lookups, never literal copy.
- Layout code uses logical `start`/`end` only ŌĆö enforced by `tool/rtl_lint.dart`.
