## Context

**The product spec.** PRD §8 (`prd.md:61`): *"KVKK/PDPL data-subject rights (export + delete self-serve)"*. PRD §10 (`prd.md:70`) makes *"KVKK/PDPL delete+export verified"* a hard MVP-release gate. `architecture.md:110`: *"self-serve export (Function → JSON email) and hard delete (cascade couple data with partner notification)"*. The M6 accept line (`implementation-plan.md:69`), verbatim: *"delete cascade integration test (couple data gone, partner notified, entitlement handled)"*. This is the last legally-required feature of the MVP.

**The hard problem is that the couple is a two-person shared object.** A couple's daily thread is jointly authored: A's answer and B's answer to the same question, revealed mutually, folded into a shared streak. When A exercises an erasure right, what happens to B's copy of that shared history is a genuine legal-and-product question the governing docs raise but never resolve (`architecture.md:110` says *notify* the partner, not *ask* them). Decision 1 resolves it. The DV threat model that shaped ADR-016 D7 and ADR-018 binds here too, in both directions: the deleting user may be a victim escaping (deletion must be total, immediate, and not vetoable), and the surviving partner may be a victim losing data (the loss must be honest, bounded, and notified).

**What the codebase gives us (Session 021 scout findings):**

- **The collection tree is a closed set of 9 paths** (`firestore.rules:16-207`, deny-by-default catch-all `:209-211`): `users/{uid}`, `users/{uid}/soloAnswers/{dayKey}`, `couples/{cid}`, `couples/{cid}/days/{dayKey}`, `couples/{cid}/days/{dayKey}/answers/{authorUid}`, `subscriptions/{cid}`, `coachUsage/{cid}`, `coachUsage/{cid}/daily/{uid}`, `invites/{code}`. Plus the Firebase **Auth record** (displayName/email/photoURL) outside Firestore. Coach chat is ephemeral (ADR-017) — no content collection exists.
- **Couple answers are stored per-author** — `answers/{authorUid}`, doc id = the author's uid (`firestore.rules:128-155`). A-authored and B-authored content is mechanically separable inside the shared thread.
- **Every client delete is already `if false`** (`firestore.rules:38,61,88,103,154,172,190,194,206,210`), each with a doc-comment reserving deletion for "the M6 KVKK/PDPL cascade Function (admin SDK)". The cascade was pre-wired from M2; this ADR makes the promise good. **No client rule is loosened.**
- **B's `users/{B}.coupleId` is client-frozen** (`firestore.rules:33-35`) — only the admin SDK can clear it. After the couple doc is deleted, every couple-scoped read B makes fails **permission-denied, not not-found** (tested at `firestore-rules.test.ts:506-513,810-818,861-866`). So the cascade's ordering decides whether B's morning-after is honest or a phantom-couple error screen (Decision 3).
- **The entitlement mirror is `subscriptions/{coupleId}`** — couple-keyed; the payer's uid appears only as a `lanes.<uid>` map key (`entitlement-core.ts:71-98`). "One purchase entitles both" is one line: `entitled: lanes.some(l => l.entitled)` (`entitlement-core.ts:603`). Absent doc = free tier (ADR-013 D5). After A's `users/{A}` doc is gone, no future RC event for A can resolve (`resolveCouple` skips missing user docs → `unresolvable` → 200 no-op, `entitlement-service.ts:82-92`) — the mirror freezes regardless of what the cascade does.
- **`resolveDiscreet(contentLanguage)` is the documented override seam** (`recipients.ts:13-20`): *"A future per-user notification-privacy SETTING will thread through this same resolver seam … keeping that override a one-function change."* Exactly two call sites (`reveal-service.ts:336`, `at-risk.ts:97`), both already holding `userData`. No per-user field exists yet.
- **`createInvite` deliberately proceeds when the caller's users doc is absent** (`invite-service.ts:71-78`: "issuing does not itself require a profile; only pairing does") and runs on the admin SDK, bypassing rules — load-bearing for the token-window analysis in Decision 2 (review finding INVITE-1).
- **The settings screen is the designated home** (ADR-018 D7, `settings_screen.dart:26-38`): "KVKK export/delete rows land HERE and EXTEND this screen." `showDialog` is legitimate there (behind the Navigator, unlike the lock screen). **Its auth-loss self-pop** (`settings_screen.dart:211-215`: `next is! AuthSignedIn` → `popUntil(isFirst)`) **also pops any route pushed above it** — load-bearing for Decision 7's error handling (review finding APP-1).
- **The sign-out path the deletion must land on** (ADR-018 / scout): the root listener `app.dart:89-96` wipes the lock on `next is AuthSignedOut`; `AuthSignedOut` is value-equal, so only a transition from a value-inequal state (here: `AuthSignedIn`) notifies. `AuthController.signOut()` (`auth_controller.dart:85-101`) is the shape to mirror: manual-op gate, explicit `state = const AuthSignedOut()` on success, `AuthError` on throw (protection stays).
- **The users update rule has no `hasOnly`** (`firestore.rules:33-35`) — a client can today write arbitrary NEW fields on its own doc; only `createdAt`/`coupleId` are frozen. Any field this ADR wants to be server-truth must gain its own freeze clause + mutation test (Decision 6, Decision 3).
- **`admin.auth().deleteUser(uid)` (singular) throws `auth/user-not-found` on a missing uid; the plural `deleteUsers(uids)` is documented idempotent** ("Deleting a non-existing user won't generate an error" — firebase-admin 13.10.0, `base-auth.js:344-349`; the repo's own test helpers already `.catch()` the singular form, `invite-preview.test.ts:44-48`). Load-bearing for Decision 2 step 7 (review findings AUTH-1/AUTH-3).
- **BulkWriter's `close()`/`flush()` never reject** — per-write failures are conveyed only via the individual op promises (`@google-cloud/firestore` `bulk-writer.d.ts:431-460`); `recursiveDelete` gives no partial-failure signal. Load-bearing for Decision 2's sweep contract (review finding SWEEP-2).
- **Test machinery:** in-process service suites seed answer docs on the trigger-isolated `demo-hayati-notrigger` project (`functions/test/support/admin.ts`); rules mutations are in-memory string surgery with anchor-rot guards (`firestore-rules.test.ts:931-1660`). Coverage gates: functions 80 hard / 85 target, app 68.

**Decision clusters that came due (the ADR-013 scaffold):**
1. What "delete everything" MEANS for the two-person shared object (Decision 1).
2. The cascade mechanism: ordering, idempotency, resumability, the mid-cascade race surface (Decision 2).
3. The partner-notification contract: channel, content, B's morning-after state (Decision 3).
4. Entitlement handling — what "handled" means at the mirror level (Decision 4).
5. Export: scope, format, delivery (Decision 5).
6. The discreet-notification override — field, write path, resolver (Decision 6).
7. The app half: flow, confirmation honesty, the lock invariants (Decision 7).
8. Fail directions (Decision 8).

**Alternatives weighed at the top level:**
- *Author-split deletion (delete only A's authored halves; tombstone the couple; B keeps B's own answers).* Rejected — Decision 1 records the full argument and the honest costs of both options.
- *Soft delete with a grace window (7–30 days).* Rejected: `architecture.md:110` prescribes hard delete; a "deleted" state that is quietly recoverable is a guarantee-vs-mechanism inversion (the app would SAY deleted while the data sits restorable); and in the DV frame a restore path is a weapon — a victim who deletes to escape must not leave a lever that the person holding their phone, SIM, or Apple ID can pull to bring the thread back. Accidental-tap protection is carried by the two-step confirmation + PIN re-verify instead (Decision 7). KVKK's "without undue delay" does not require a cooling-off window.
- *Email delivery for export.* Rejected: needs an email provider = a NEW operator dependency the resume-prompt tells us to avoid; and an emailed JSON is a plaintext copy of relationship data sitting in an inbox the partner may share. In-app delivery is both cheaper and more protective (Decision 5).
- *Blocking deletion on partner consent.* Rejected without hesitation: a consent gate hands an abusive partner a veto over the victim's erasure right. Notification, never consent (`architecture.md:110` already says exactly this).
- *A `coupleEnded` push notification.* Adopted in rev 1, **deleted in rev 2** (review finding DV-1, adjudicator-confirmed over a skeptic downgrade — the governing-docs verdict wins per the S018 rule): the governing mandate ("partner notification") names no channel and no timeliness, and the ADR's own text already made the Firestore field the guarantee carrier — so the push was pure added risk: a proactive, real-time ping to a possibly-abusive partner at the deleting victim's moment of escape, the documented peak-danger moment. Worse, for an AR-locale B the discreet fallback would deliver the terminal event as the invitational lure "Something is waiting for you in Hayati." (`payload-policy.ts:35-39`) — an engagement come-on for a deletion notice, dishonest on its face (DV-1's verified deeper defect). Decision 3 is field + in-app notice only; a push can be added later by a new decision if the founder wants one.
- *`beforeUserDeleted` blocking Auth trigger + client-side `currentUser.delete()`.* Rejected: blocking Auth triggers require the Identity Platform upgrade; and client-driven auth deletion hits `requires-recent-login`, putting a re-auth wall in front of a legal right. A callable that does everything server-side (auth deletion last) needs neither.

## Decision 1 — "Delete everything" means the couple is terminal: the shared thread dies with the couple, both halves; each partner's solo space is theirs alone

- **What A's deletion removes, exactly:** A's `users/{A}` + all `soloAnswers`; A's Firebase Auth record; every invite naming A (`creatorUid == A` or `joinerUid == A`); **the entire couple subtree** — `couples/{cid}`, every `days/{dayKey}`, every `answers/{authorUid}` doc **for both authors**, `subscriptions/{cid}` (Decision 4), `coachUsage/{cid}` and both `daily/{uid}` lanes; and the transient `deletions/{A}` progress doc (Decision 2). What survives: B's `users/{B}` doc (minus the `coupleId` pointer, plus the `coupleEnded` marker — Decision 3), B's `soloAnswers`, B's Auth record.
- **Why the whole thread and not just A's half.** Three arguments, in decreasing order of weight:
  1. **B's answers in the shared thread are mixed personal data.** They were written *to A, about the relationship with A*, inside a mutual-reveal structure; they routinely describe A. An erasure right that leaves the other side of every conversation readable by the partner has not erased the relationship record — it has redacted half of it. In the DV frame this is decisive: a victim who deletes to escape must not leave the abuser a curated archive of the relationship.
  2. **The governing docs already say it.** The accept line is *"couple data gone"* (`implementation-plan.md:69`) and `architecture.md:110` says *"cascade couple data"* — not "cascade the requester's half."
  3. **A half-thread is dishonest data.** B's answers were revealed *in exchange for* A's; stripped of their counterparts they misrepresent what B agreed to share and with whom.
- **The alternative, recorded loudly (the founder can overturn this with a superseding ADR):** *author-split deletion* — delete only `answers/{A}`, `daily/{A}`, `lanes[A]`; keep a tombstoned couple (`memberUids` scrubbed to `[B]` or a status flag) so B retains read access to B's own authored answers. Honest costs of the split: B keeps content that is substantially *about A* after A asked for erasure; the tombstone couple is a permanent ghost object needing new rules, new states, and a re-pairing story; and the app must explain to B why half of every day is missing — a worse product experience than a clean, honest ending. Honest cost of the chosen option: **B loses B's own authored couple-thread content without B's consent, and gets no pre-deletion preservation window or export opportunity** (review finding DV-3 — recorded here, in the D8 table, and in Consequences: a preservation window for B is structurally a delay/veto on A's erasure, and Decision 1's DV frame rejects that trade; B can export B's own surviving data — profile, solo answers — at any time via Decision 5). We say the loss to B plainly in the notification copy (Decision 3) rather than hiding it. B's solo reflections — the space that was only ever B's — are untouched.
- **What no mechanism can carry (stated so nobody claims it):** deletion removes the server copies. It cannot un-reveal what B already read, screenshot, or remembers. The confirmation copy must not promise otherwise (Decision 7 pins the copy to "deleted from Hayati", never "your partner will no longer have seen it").
- **Backups/PITR residual:** no deploy has ever happened and no PITR/backup schedule exists today, so "deleted" is currently literal. When the deploy era configures operational backups, backup-cycle retention must be aligned and documented (KVKK tolerates documented backup-cycle deletion delays). Recorded here so the first deploy session inherits it; not buildable now.
- **Deletion is per-user, not per-couple, on the requester's side:** an unpaired A (no `coupleId`) gets the same flow minus the couple steps — profile, solo answers, invites, Auth record. Test-pinned.

## Decision 2 — One idempotent, resumable cascade callable: `deleteAccount`; the cursor is authoritative, sweeps confirm-or-throw, auth deletion is last and idempotent

- **The Function:** `deleteAccount` — `onCall`, `europe-west1`, `enforceAppCheck: false` (matches every callable), auth-guarded, **`timeoutSeconds: 540`** (it owns the only unbounded multi-collection work in the codebase; the 60s default is a mis-fit — review finding TIMEOUT-5). The request must carry `confirm: 'DELETE'` (a wire-level literal sent by the app, never typed by the user — no localization or screen-reader surface; anything else → `invalid-argument`) so no client bug can invoke deletion by accident. Response: `{ status: 'deleted' }`. Errors follow the coach taxonomy: static messages, typed `details.reason`, **no content and no uids in any log line** (ADR-016 D8 / `architecture.md:110` discipline — deletion events are special-category-adjacent: log step names and counts, never identifiers beyond what Functions logging already stamps).
- **The cursor is authoritative (review finding CASCADE-1, blocking — the rev-1 text gated cursor use on "users/{A} gone", which never fires in the exact window the cursor exists for: after the detach clears `coupleId` while `users/{A}` still exists, a re-drive would have misread A as unpaired and orphaned the entire couple subtree forever, then deleted the last pointer to it).** Rev 2 rule: **step 1 consults `deletions/{A}` FIRST; whenever the cursor exists, its recorded `coupleId`/`partnerUid` drive the couple steps.** The live `users/{A}.coupleId` read is only the *initial* resolution used to WRITE the cursor on the first run. The cursor is written before any destructive step and removed only after every destructive Firestore step has confirmed — so every kill window re-drives through it.
- **The pinned order (each step idempotent; re-drive = re-run from the top, completed steps no-op):**
  1. **Resolve + record.** Read `deletions/{A}`: if present, adopt its `coupleId`/`partnerUid` verbatim (never re-derive). If absent, read `users/{A}.coupleId`; if paired, read `couples/{cid}.memberUids` → partnerUid; merge-write `deletions/{A}` `{ coupleId, partnerUid, startedAt }` (or `{ coupleId: null }` for an unpaired A). If both `deletions/{A}` and `users/{A}` are absent (a prior run died after step 6), skip to step 7.
  2. **The detach transaction** (single Firestore txn, skipped when `coupleId` is null or `couples/{cid}` is already gone): delete `couples/{cid}`; on `users/{B}`: delete `coupleId` + set `coupleEnded: { at: serverTimestamp }`; on `users/{A}`: delete `coupleId`. **Guard: B is touched only if `users/{B}.coupleId == cid`** — on a re-drive after B has already re-paired, B's new couple must not be dented (test-pinned). Deleting the couple doc FIRST is the ordering lever: it instantly fail-closes both members' couple-scoped read/write surface (rules `get()` on the couple now denies everything — the mid-cascade race answer), while B's honest re-route rides the atomic `coupleId` clear, never a phantom-couple `permission-denied`.
  3. **Couple-subtree sweep** (admin list-and-delete; orphaned subcollections remain listable after the parent doc's deletion): every `days/{dayKey}/answers/*`, every `days/{dayKey}`, `coachUsage/{cid}/daily/*`, `coachUsage/{cid}`, `subscriptions/{cid}`.
  4. **Invites sweep:** query `invites` on `creatorUid == A`, then `joinerUid == A` (both single-field equality queries — auto-indexed); delete all hits (consumed invites' audit value yields to the erasure mandate; the couple they document is gone).
  5. **Own-subtree sweep:** every `users/{A}/soloAnswers/*`, then `users/{A}` itself.
  6. **Remove the cursor:** delete `deletions/{A}` — only reachable after steps 2–5 confirmed zero residue.
  7. **Delete the Auth user — idempotently** (review findings AUTH-1 + AUTH-3, the latter adjudicator-upgraded to blocking: the singular `admin.auth().deleteUser` throws `auth/user-not-found` on an already-deleted uid, so the rev-1 "re-attempts only step 8" re-drive would THROW on a lost-response retry, telling the user deletion failed on an account that is fully erased — and after the token hour, nobody could ever re-drive). Rev 2 pins the mechanism: **use `admin.auth().deleteUsers([A])` (documented idempotent) and treat a `user-not-found` failure entry as success; any other failure entry throws.** Auth deletion stays **deliberately last**, so any earlier failure leaves A authenticated and able to re-drive the callable.
- **The sweep contract: confirm-or-throw (review finding SWEEP-2 — BulkWriter's `close()` never rejects; per-write failures surface only on the individual op promises, and `recursiveDelete` has no partial-failure signal at all; a swallowed delete + auth-deleted-last = permanent, success-reporting residue).** Pinned for **every** sweep step (3, 4, 5 — the verifiers' deeper finding: step 5's soloAnswers residue would be exactly as permanent): every delete's op promise is awaited (or `onWriteError` rethrows after retries); **any unconfirmed delete aborts the callable before step 7**; bare `recursiveDelete` and bare `await close()` are forbidden by test (the fault-injection suite drives a failing delete and asserts the callable throws and Auth survives). `{ status: 'deleted' }` is returned only after zero confirmed residue.
- **Kill-mid-cascade proof (the firm deliverable):** the service takes an injectable `checkpoint(step)` hook (the `deps.beforeWrite` prior art from `reveal-service.ts:47-58`), **including an explicit checkpoint around the non-transactional step 7** (AUTH-1's deeper finding: a Firestore-write-shaped seam cannot naturally express "kill after the Auth call"); the test loop kills the cascade after each step k — **k=2 and k=3 are the windows CASCADE-1 proved fatal in rev 1 and get named assertions that the couple subtree is swept on re-drive** — re-drives, and asserts the full postcondition set converges. Concurrent double-delete (both partners simultaneously) converges by the same idempotency.
- **The token-window residual is a CLASS over two surfaces, not a list of two writes (review finding INVITE-1 — rev 1 analyzed only the direct-client-write surface).** A deleted user's ID token stays valid up to ~1h (rules and callables verify the token, not account existence). Surface one, **direct client writes** (rules-governed): the couple surface is fail-closed at detach; `soloAnswers` writes gain an `exists(users/$(uid))` rules guard (mutation-tested — Decision 6 carries the rules diff); a re-created `users/{A}` profile doc remains possible and accepted (an unreadable-forever orphan, equivalent to a fresh sign-up, requiring a deliberately hostile client — named, not silent). Surface two, **admin-SDK callables** (rules-blind — a rules guard is structurally incapable of covering them): `createInvite` today proceeds without a users doc and would mint a fresh, never-swept invite naming the deleted uid (`invite-service.ts:71-78,124-129`); **rev 2 closes it: `issueInvite` gains a creator-profile-must-exist precondition (absent `users/{creator}` → `failed-precondition`), handler-tested.** The remaining callables are verified self-closing: `joinInvite` throws profile-missing, `coachProxy` fails membership, `exportData` and `updateNotificationPrivacy` gain the same profile-exists precondition (`update()` on a missing doc fails natively; pinned by test, not assumed). The class framing is the fix: any FUTURE auth-only callable must state which side of this boundary it is on.
- **New-writer governance:** this Function is a new admin-SDK writer over collections whose doc-comments already anticipate exactly it. `revenueCatWebhook` remains the sole writer of `subscriptions/{cid}` **content**; the cascade only ever deletes that doc whole (never edits lanes) — the ADR-013 sole-writer claim narrows to "sole content writer," recorded in the §3 docs pass.

## Decision 3 — Partner notification: a server-truth field written atomically with the detach, rendered as an honest terminal screen in-app; deliberately NO push

- **The channel is Firestore, and only Firestore.** `users/{B}.coupleEnded = { at: Timestamp }` is written in the detach transaction (Decision 2 step 2) — atomic with the moment B's couple ceases to exist, present on every device B signs into, and deliverable today. It carries **no reason, no message, and no reference to A** — B's client needs none of those to render the honest state, and A's identifiers on B's doc would be residue the cascade just promised to remove.
- **No push, by decision (review finding DV-1 — the skeptic downgraded, the governing-docs adjudicator confirmed; per the S018 rule the adjudicator's verdict stands).** The governing mandate ("partner notification", `architecture.md:110`; "partner notified", `implementation-plan.md:69`) names no channel and no timeliness, and rev 1's own text already made the field the guarantee carrier — so the push added only *proactive real-time delivery to a possibly-abusive partner at the deleting victim's moment of escape*, the documented peak-danger moment in the DV literature this product's threat model takes seriously. The verified deeper defect sealed it: for an AR-locale B (discreet default ON), the push would have arrived as the invitational lure *"Something is waiting for you in Hayati."* — an engagement come-on announcing a terminal deletion, dishonest on its face. **"Partner notified" therefore means: the atomic server field + the in-app notice screen B reaches on their next open.** B learns when B comes to the app, not when A flees it. If the founder ever wants a proactive push, that is a new decision against this recorded analysis, riding the existing ADR-012 seams (which remain untouched — no new `PushKind`, no `payload-policy.ts` change, no send latch).
- **Field governance:** `coupleEnded` is server-truth: the users update rule gains a freeze clause (same shape as the `coupleId` freeze; clients can neither mint nor clear it), mutation-tested. It is cleared by the next successful `joinInvite` (the join transaction deletes it from both members — a one-line extension), so a re-pairing B does not carry a stale tombstone.
- **The "seen" state is keyed by the event, not by the user (review findings NOTICE-1 + DV-1-app — a single set-once flag on the never-cleared `LocalFlagStore` would swallow the notice for a SECOND `coupleEnded` after a re-pair: the exact wordless vanish this Decision exists to prevent).** The device-local seen flag's key embeds the event timestamp: `coupleEndedSeen.<uid>.<at.millisecondsSinceEpoch>`. A new `coupleEnded` (necessarily a different `at`) mints a fresh key, so every ending is noticed exactly once per device; set-once semantics per key are exactly right. Per-uid keying keeps the flag from leaking across accounts on a shared device (the coach-ack precedent).
- **B's morning-after state (the honest terminal screen):** the onboarding gate today routes `coupleId == null` → solo home silently (`onboarding_gate.dart:59-66`) — which would make B's couple *vanish wordlessly*, a dishonest state. The gate gains one branch, **evaluated before the pending-invite/deep-link branch** (review finding NOTICE-2 — a joiner arriving by deep link with `coupleEnded` set must see the notice first; the invite flow continues after Continue): `coupleId == null && coupleEnded != null && !seen(key)` → render `CoupleEndedNoticeScreen` **as a gate-rendered child, not a pushed route** (review finding APP-2 — the gate cannot be popped back to; Continue sets the flag and bumps a small provider — `coupleEndedSeenProvider`, a keepAlive notifier seeded from `LocalFlagStore` at gate build — whose change re-evaluates the gate; pinned as mechanism, not left to "the gate notices"). **The app-side wire mapping is pinned** (review finding APP-3): `ProfileDto`/`RelationshipProfile` gain a `coupleEndedAt: DateTime?` mapped from the nested `{at: Timestamp}` — the existing mapper does not convert nested maps by magic, and the field names are wire contract.
- **The notice copy (×3 locales, native review flagged) attributes no actor (review finding DV-2 — "your partner deleted this" both blames and strips the escaping deleter of deniability):** the shared space **has been closed** and its content — both sides of it — permanently deleted; your own private reflections are untouched and remain yours; if you had a subscription, it is not cancelled by this — manage it in your App Store settings; you can pair again whenever you choose. One primary action ("Continue") → sets the event-keyed seen flag → solo home. Only A could in fact have done it, and B will usually know — but the app does not say it, and that reticence is deliberate and recorded.
- **The race at the deletion instant:** if B is ON the paired home, the profile stream (which the gate already watches) emits the cleared `coupleId` and re-routes live — usually before the orphaned couple listener can even error; the existing `_PairedErrorView` remains the fallback for the losing side of that race, and the notice renders on the next gate evaluation. No crash, no ghost couple — the accept line's words.

## Decision 4 — Entitlement "handled" = the mirror dies with the couple (ADR-013 Option A); the real subscription is out of reach and we say so

- **The cascade deletes `subscriptions/{cid}` whole** in the subtree sweep. "Absent doc = free tier" is already the defined read model (ADR-013 D5), so this is a well-defined downgrade, not an undefined state. Both members read free immediately (B via the honest notice; B's couple is gone anyway — there is no surviving couple context to be premium).
- **Alternatives rejected:** *lane-scrub + re-derive* (ADR-015-consistent only if a single-member couple survives, which Decision 1 rejected); *tombstone the payer's lane* (couples the cascade to the webhook's LWW order-key model for zero benefit once the doc is deleted anyway); *do nothing* (leaves an orphan uid-bearing doc — data residue in a flow whose purpose is erasure — plus a paid-but-downgraded freeze window).
- **What this deliberately does NOT claim:** the cascade cannot cancel, refund, or even see the real RevenueCat/App Store subscription — those are RC *account* operations behind the secret key that does not exist until operator item 0 (the stopping condition in the resume-prompt anticipated exactly this boundary; the RC-REST reconciliation debt already rides item 0 + the deploy era). Consequences, named per payer:
  - **A (the deleter) was the payer:** Apple keeps billing A's Apple ID until A cancels in App Store settings. A's confirmation screen says so in plain words (Decision 7) — deletion of the account is not cancellation of the subscription, and pretending otherwise would be the exact over-claim class S020 taught us to delete.
  - **B (the survivor) was the payer:** B pays for a couple that no longer exists. B's notice screen carries the same static App-Store-settings line (static for everyone — the copy never reveals *who* paid, so it leaks nothing). If B re-pairs, the next RC lifecycle event resolves through `users/{B}.coupleId` and mints the mirror on the new couple (`resolveCouple` mechanism, no new code) — stated as mechanism, not promised as UX (an annual subscription's next event can be months away; the reconciliation debt covers the gap when it lands).
- **Webhook aftermath:** post-cascade RC events for A resolve to nothing → `unresolvable` → loud 200 skip (`entitlement-service.ts:234-241`) — already the designed behavior, no code change, no error noise.

## Decision 5 — Export: one callable returning a versioned JSON document, delivered in-app; strictly the requester's own data; no premium gate; no email provider

- **The Function:** `exportData` — `onCall`, auth-guarded, requester-profile-must-exist (`failed-precondition` otherwise — the Decision 2 callable-surface rule), **no premium gate and no cap** (a data-subject right is not a feature tier; personal-use scale makes abuse a deploy-era concern, recorded with the existing rate-limiter hardening note). Returns the export document directly in the callable response.
- **Scope — everything A authored or that is about A, nothing the partner did (the firm deliverable's words):**
  - `profile`: `users/{A}` client fields (`status`, `contentLanguage`, `register`, `createdAt`) + the Auth record (`displayName`, `email`, `photoURL`) + `notificationPrivacy` if set.
  - `soloAnswers[]`: dayKey, questionId, question text (app content, included for readability), answer text, answeredAt.
  - `coupleContext` (when paired): `coupleId`, `createdAt`, `timezone`, `streak` (jointly derived, member-readable by design), the coach monthly counter (member-readable by design). **`memberUids` and every other appearance of B's uid are excluded** — the partner's identifier is the partner's data.
  - `coupleAnswers[]`: **only `answers/{A}`** — dayKey, questionId, question text, A's answer text, answeredAt, whether that day revealed. **Never B's text — and the honest rationale is B-data-minimization over a two-subject object, not a side channel (review finding EXPORT-1, skeptic-refuted / adjudicator-downgraded to this wording fix):** for *revealed* days A can already read B's answers live in-app (`firestore.rules:144-146`), so excluding them from the export is deliberately delivering LESS than A's in-app view — a durable, forwardable copy of B's words is a different artifact from a couple-scoped read, and the repo's recorded DV-minimization standard (ADR-016 D7) decides the tension in favor of B. The export screen's copy says "your data", and means it. (The exclusion *also* closes the pre-reveal side channel, which is the only day-class where "side channel" is the accurate word.)
  - `coachUsage`: A's own `daily/{A}` lanes only — the ADR-016 D7 self-read invariant applied to export verbatim; B's usage frequency never crosses.
  - `subscription`: the summary + `lanes[A]` only.
  - `invites[]`: codes A created or redeemed — code, status, timestamps; **the counterpart uid scrubbed**.
  - Envelope: `{ formatVersion: 1, generatedAt, uid, data: {…} }`. Field names are a wire contract (the profile enum-`name` precedent, `architecture.md:72`); `formatVersion` exists so a future shape change is explicit.
- **Delivery: in-app.** The app renders the document on an export screen (selectable, copy-to-clipboard). No email provider, **no new operator item** — this is the deliberate reversal of the "email" parenthetical at BOTH its occurrences (see the Supersedes note). A share-sheet/file export is a recorded future nicety, not v1.
- **Size honesty:** at this product's shape (≤2000-char answers, one per person per day), years of use fit in tens-to-hundreds of KB against a ~10MB callable ceiling. The bound is real but distant; if it is ever approached, the recorded upgrade path is pagination or a signed Storage URL — a new ADR, not a silent stretch.
- **What export deliberately does NOT include:** B's anything (stated above); coach conversation content (none exists — ADR-017 ephemeral; if operator item 7 ever persists `coach_sessions`, its self-read-only bundle explicitly includes M6 export/cascade integration — recorded, not silently omitted); the device-local Keychain lock record (device data, wiped by sign-out per ADR-018, never server-resident); crisis/help-path linkage (ADR-016 D8 — no such join exists server-side and the export must never reconstruct one).

## Decision 6 — The discreet-notification override: `users/{uid}.notificationPrivacy`, written by a settings Function, read through the `resolveDiscreet` seam; the complete rules diff lives here

- **The field:** `users/{uid}.notificationPrivacy: 'discreet'` — a string enum with exactly one legal value in v1, absent meaning "locale default". An enum rather than a boolean so a future `'named'` value (an AR-locale user opting OUT of the discreet default) is representable without migration. **v1 is deliberately opt-in-only**: the AR default (PRD F6) stays non-overridable — weakening a protective default is a separate product decision, recorded here, not smuggled in.
- **The write path:** callable `updateNotificationPrivacy({ discreet: bool })` — sets the field or deletes it, via `update()` on the existing doc so an absent profile fails natively (the Decision 2 callable-surface rule, test-pinned). Server-validated; the users-doc client-write surface stays exactly as it was. This follows the M6.1 deferral's own words ("a `users` field + a settings Function") — and it is made TRUE by a rules change: the users update rule gains a `notificationPrivacy` freeze clause (the `coupleId` freeze shape), so "server-owned field" is a mechanism, not a sentence. Without the freeze, the claim would outrun the rules (the users update rule has no `hasOnly` — scout finding), which is precisely the S020 finding class.
- **The read path:** `resolveDiscreet(contentLanguage, notificationPrivacy)` → `notificationPrivacy === 'discreet' || contentLanguage === 'ar'`. Junk values are ignored (the `contentLanguageOf` defensive-reader pattern). Both call sites pass the field from the `userData` they already hold — the one-function-plus-two-args change the seam comment promised.
- **The settings row:** a `SwitchListTile` on the settings screen — value = the explicit field (streamed via the existing profile watch; `ProfileDto` maps the new field), busy/error per the `_ErrorLine` idiom. For AR-locale users the subtitle states the default is already on; flipping the switch on writes the explicit value anyway (deliberately: an AR user who later changes `contentLanguage` keeps their discreet posture instead of silently losing it — the honest reason the explicit write matters).
- **This session's complete rules diff (all mutation-tested, nothing else moves):**
  1. users update: freeze `notificationPrivacy` (client write of it must fail; weakening the clause must re-admit the write). Same clause shape for `coupleEnded` (Decision 3).
  2. `soloAnswers` create/update: add `exists(/databases/$(database)/documents/users/$(uid))` — the deleted-user orphan-write guard for the direct-write surface (Decision 2; the callable surface is guarded in the Functions, where rules cannot reach). Positive control: a normal write with a live profile still succeeds; mutation: weaken → the orphan write succeeds.
  - Every existing `allow delete: if false` stays byte-identical; the cascade needs zero loosening (admin SDK bypasses rules — scout-verified, already mutation-pinned). New delete-deny mutants for `users`/`soloAnswers`/`couples` (the three the harness notes as uncovered) are added as test-coverage, not rules changes.

## Decision 7 — The app half: two rows + a toggle on the existing settings screen; the server cascade is a local operation, only the teardown touches auth state; the four lock invariants untouched

- **Surface:** the M6.1 `SettingsScreen` gains "Download my data", "Delete account & data", and the Decision-6 toggle — extending the existing `ListView` (ADR-018 D7's explicit reservation; no second surface). Everything stays behind the lock like the rest of the app — deliberately (resume-prompt handoff): a pre-auth/pre-unlock deletion entry point would be its own ADR.
- **The export flow:** row → `ExportScreen` (pushed, the `PinSetupScreen` mold) → calls the repository → renders the JSON with copy-to-clipboard; loading and honest error states per the coach taxonomy mapping. A "Download your data first?" link sits inside the deletion flow, one tap away.
- **The deletion flow — never a one-tap accident, and honest to the letter:**
  1. Row → `DeleteAccountScreen` (pushed) stating, in plain sentences ×3 locales (native review flagged): this is **irreversible**; it deletes your account, your solo reflections, and the **entire shared space — both sides of every answer** — permanently; your partner will see that the space was closed (not why); **it does not cancel any App Store subscription** — manage that in your App Store settings; consider downloading your data first. The copy never claims the partner "loses what they've already seen" (Decision 1's honesty bound).
  2. Confirm → if the lock is enabled, `_PinVerifyDialog` (the existing seam — attempt-bounded, cooldown-aware; any biometric anywhere stays behind the `biometricOnly` sentinel, untouched); if no lock, a destructive-styled confirmation dialog naming the word "permanently". `showDialog` is legitimate here — settings is behind the Navigator (Invariant D applies only to lock-screen children).
  3. **Two phases with different state owners (review finding APP-1, confirmed serious — rev 1 mapped a server-cascade failure to global `AuthError`, and the host SettingsScreen's auth-loss self-pop (`settings_screen.dart:211-215`, fires on any `next is! AuthSignedIn`) would have popped the DeleteAccountScreen before its promised "honest retry copy" could ever render, and made the belt-and-suspenders wipe dead code):**
     - **Phase 1 — the server cascade — is a local operation.** `AuthController.deleteAccount()` holds the manual-op gate for the whole operation, but a *callable* failure is caught, the state is left exactly as it was (`AuthSignedIn` — nothing transitions, nothing pops), and a typed exception is rethrown for `DeleteAccountScreen` to render retry copy in place. Re-driving is safe by Decision 2. A retry that returns `deleted` because the previous attempt actually completed (the lost-ack case) is indistinguishable from success — the screen's error copy therefore says the deletion "could not be confirmed", never that it "failed" (AUTH-3's skeptic-verified residual: a "failed" message on a completed deletion would be a lie).
     - **Phase 2 — session teardown — only after server success.** The Google session half is attempted and swallowed (a live Google session for a deleted account is meaningless residue, not a protection); `_auth.signOut()` runs; on success the controller sets `state = const AuthSignedOut()` explicitly. The pre-state is `AuthSignedIn`, so the transition is value-inequal and the root listener (`app.dart:89-96`) fires the lock `wipe()` — the resume-prompt's binding handoff, satisfied on the existing path.
     - **If phase 2 throws:** `AuthError`. The host self-pop dumps to the root shell and the lock stays — acceptable and honest, because retry is unnecessary (the deletion is complete) and the state self-heals: the dead session's next token refresh fails (≤~1h) → the auth stream emits null → `AuthSignedOut` → the root listener wipes (the D8 row-5 correction, from AUTH-3's adjudicator: ADR-018's protection-stays asymmetry must not be misapplied to a success masquerading as an error — here the *protective* thing and the *self-healing* thing are the same path, and we do not pretend the popped screen still shows retry copy).
  4. **The root listener is the sole wipe carrier on success (APP-1's second half, stated honestly):** rev 1 claimed a belt-and-suspenders settled-state wipe on the delete screen; that screen is popped by the host listener at the exact transition it would need to observe, so the claim was false redundancy and is deleted. The root listener + the recovery path's settled-state wipe (for the orphaned-record boot, SPEC-1) are the real carriers. **Process death mid-flow** (review finding LOCK-1): if the process dies after the server deleted Auth but before teardown, the next boot either restores a dead session (token refresh fails → stream null → transition → wipe) or boots straight into `AuthSignedOut` with a lock record present — the ADR-018 orphaned-record recovery path ("Forgot PIN? Sign out" reads the settled state and wipes) already covers exactly that state. Named, mechanism verified, nothing new to build.
- **Invariant audit:** (A) nothing invalidates `privacyLockControllerProvider` — the wipe is reached via the root listener (and the existing recovery path); (B) no new mutating method is added to the lock controller at all; (C) `biometricOnly` untouched — deletion re-auth is PIN-only through the existing dialog; (D) no dialogs above the Navigator — all new UI is pushed routes/dialogs below the gate.
- **Free tier untouched, assertion-proven:** the two rows + toggle render for free users (a data right is not premium); the daily loop and every pre-existing golden stay byte-identical (the M5.2 probe-pin precedent).

## Decision 8 — Fail-direction rules (the table reviewers check first)

| Failure / vector | Direction | Why |
|---|---|---|
| Cascade dies mid-run (any step) | **Resumable** — callable errors honestly; A stays authenticated (auth delete is last); re-drive converges through the authoritative cursor | never a permanently half-deleted couple (D2, CASCADE-1 fix) |
| Sweep delete unconfirmed (BulkWriter partial failure) | **Callable throws before auth delete** — confirm-or-throw pinned for steps 3/4/5 | success-with-residue is the silent-failure class (D2, SWEEP-2 fix) |
| Step-7 retry on an already-deleted Auth user | **No-op success** — `deleteUsers([A])`, `user-not-found` = done | a completed deletion must never report failure (D2, AUTH-1/AUTH-3 fix) |
| Re-drive after B re-paired | B's new couple **untouched** — detach guard `users/{B}.coupleId == cid` | a stale cascade must not dent a new relationship (D2) |
| Both partners delete concurrently | **Converges** — idempotent deletes, per-uid cursors | (D2) |
| Server cascade fails (phase 1) | **State untouched, screen survives, retry in place** — typed exception, never `AuthError` | the host self-pop must not eat the retry surface (D7, APP-1 fix) |
| Server delete complete, local sign-out throws (phase 2) | `AuthError`; lock stays; **self-heals** to `AuthSignedOut` on the dead session's token-refresh failure (≤~1h) or manual sign-out | protection-stays must not strand a completed deletion (D7, AUTH-3 fix) |
| Process death mid-flow after auth deleted | Next boot: stream-null transition wipes, or the SPEC-1 orphaned-record recovery path | existing mechanisms, named (D7, LOCK-1) |
| Deleted user's still-valid token (≤~1h) | Couple surface **fail-closed** at detach; `soloAnswers` **fail-closed** by the rules exists-guard; callables guarded by profile-exists preconditions (`createInvite` tightened); `users/{A}` re-create = recorded residual | the residual is a class over both surfaces (D2, INVITE-1 fix) |
| RC event for the deleted payer arrives | `unresolvable` → loud 200 skip, no write, no error noise | existing ADR-013 D3 behavior (D4) |
| Payer's real subscription | **Out of reach — billing continues until cancelled in App Store**; said plainly in both users' copy | RC account state is behind operator item 0 (D4) |
| Export larger than the callable ceiling | Callable errors honestly; recorded upgrade path (pagination / signed URL) | distant bound, named not silent (D5) |
| B on the paired home at deletion instant | Profile stream re-routes to the notice; `_PairedErrorView` is the race fallback | no crash, no ghost couple (D3) |
| B's second couple also ends | **Notice shows again** — event-keyed seen flag | a set-once flag would swallow it (D3, NOTICE-1 fix) |
| `coupleEnded` faked or cleared by a client | **Impossible** — rules freeze, mutation-tested | server-truth field (D3, D6) |
| B never opens the app | **Unnotified until next open — accepted**: no push, by decision | the proactive ping was the DV risk (D3, DV-1) |
| B's preservation of their own thread half | **None — accepted and recorded**: a window is structurally a delay/veto on A's erasure | (D1, DV-3) |
| Backups (deploy era) | Recorded obligation: align backup retention with erasure; nothing exists today | (D1) |
