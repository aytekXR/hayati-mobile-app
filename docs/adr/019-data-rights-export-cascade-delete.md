# ADR-019: KVKK/PDPL data rights — self-serve export, hard cascade delete with partner notification, and the per-user discreet-notification override

- **Status:** Accepted (rev 1 — pre-code; the adversarial design review will fold rev 2 before implementation)
- **Date:** 2026-07-12 (Session 021)
- **Deciders:** engineering session, per PRD §8 NFR ("KVKK/PDPL data-subject rights (export + delete self-serve)") + PRD §10 release criteria ("KVKK/PDPL delete+export verified") + implementation-plan M6.2
- **Related:** ADR-005 (couple-scoped data model — bounds the cascade's scope; NB its `thread[]` day-doc shape is stale, superseded by ADR-011/012), ADR-012 (push policy — the partner notification rides `composePush`/`MessagingPort` and its no-content invariant; the `resolveDiscreet` seam this ADR finally fills), ADR-013 (entitlement mirror — "absent doc = free tier", "delete = M6 cascade" carve-out), ADR-015 (transfer tombstones — the revoke-asymmetry precedent; the lane-scrub alternative rejected here), ADR-016 D7/D8 (DV posture — self-read-only coach daily lane binds the export scope; no-content logging binds the cascade's logs), ADR-017 (coach threads ephemeral — nothing coach-content-shaped exists for the cascade to cover), ADR-018 (the settings screen this extends; the `AuthSignedOut` wipe path the deletion flow must land on), docs/architecture.md §3/§4/§8, docs/prd.md §8/§10, docs/mvp.md item 12
- **Supersedes in place:** the parenthetical export shape in `architecture.md:110` ("Function → JSON **email**") — Decision 5 replaces email delivery with in-app delivery, and §8 is updated in this session's docs pass. The erasure and notification halves of that sentence stand.

## Context

**The product spec.** PRD §8 (`prd.md:61`): *"KVKK/PDPL data-subject rights (export + delete self-serve)"*. PRD §10 (`prd.md:70`) makes *"KVKK/PDPL delete+export verified"* a hard MVP-release gate. `architecture.md:110`: *"self-serve export (Function → JSON email) and hard delete (cascade couple data with partner notification)"*. The M6 accept line (`implementation-plan.md:69`), verbatim: *"delete cascade integration test (couple data gone, partner notified, entitlement handled)"*. This is the last legally-required feature of the MVP.

**The hard problem is that the couple is a two-person shared object.** A couple's daily thread is jointly authored: A's answer and B's answer to the same question, revealed mutually, folded into a shared streak. When A exercises an erasure right, what happens to B's copy of that shared history is a genuine legal-and-product question the governing docs raise but never resolve (`architecture.md:110` says *notify* the partner, not *ask* them). Decision 1 resolves it. The DV threat model that shaped ADR-016 D7 and ADR-018 binds here too, in both directions: the deleting user may be a victim escaping (deletion must be total, immediate, and not vetoable), and the surviving partner may be a victim losing data (the loss must be honest, bounded, and notified).

**What the codebase gives us (Session 021 scout findings):**

- **The collection tree is a closed set of 9 paths** (`firestore.rules:16-207`, deny-by-default catch-all `:209-211`): `users/{uid}`, `users/{uid}/soloAnswers/{dayKey}`, `couples/{cid}`, `couples/{cid}/days/{dayKey}`, `couples/{cid}/days/{dayKey}/answers/{authorUid}`, `subscriptions/{cid}`, `coachUsage/{cid}`, `coachUsage/{cid}/daily/{uid}`, `invites/{code}`. Plus the Firebase **Auth record** (displayName/email/photoURL) outside Firestore. Coach chat is ephemeral (ADR-017) — no content collection exists.
- **Couple answers are stored per-author** — `answers/{authorUid}`, doc id = the author's uid (`firestore.rules:128-155`). A-authored and B-authored content is mechanically separable inside the shared thread.
- **Every client delete is already `if false`** (`firestore.rules:38,61,88,103,154,172,190,194,206,210`), each with a doc-comment reserving deletion for "the M6 KVKK/PDPL cascade Function (admin SDK)". The cascade was pre-wired from M2; this ADR makes the promise good. **No client rule is loosened.**
- **B's `users/{B}.coupleId` is client-frozen** (`firestore.rules:33-35`) — only the admin SDK can clear it. After the couple doc is deleted, every couple-scoped read B makes fails **permission-denied, not not-found** (tested at `firestore-rules.test.ts:506-513,810-818,861-866`). So the cascade's ordering decides whether B's morning-after is honest or a phantom-couple error screen (Decision 3).
- **The entitlement mirror is `subscriptions/{coupleId}`** — couple-keyed; the payer's uid appears only as a `lanes.<uid>` map key (`entitlement-core.ts:71-98`). "One purchase entitles both" is one line: `entitled: lanes.some(l => l.entitled)` (`entitlement-core.ts:603`). Absent doc = free tier (ADR-013 D5). After A's `users/{A}` doc is gone, no future RC event for A can resolve (`resolveCouple` skips missing user docs → `unresolvable` → 200 no-op, `entitlement-service.ts:82-92`) — the mirror freezes regardless of what the cascade does.
- **`resolveDiscreet(contentLanguage)` is the documented override seam** (`recipients.ts:13-20`): *"A future per-user notification-privacy SETTING will thread through this same resolver seam … keeping that override a one-function change."* Exactly two call sites (`reveal-service.ts:336`, `at-risk.ts:97`), both already holding `userData`. No per-user field exists yet.
- **Push kinds are a closed union** (`payload-policy.ts:20`): `partnerAnswered | reveal | streakAtRisk`. Copy lives inline in `payload-policy.ts` ×3 languages; the discreet branch ignores kind entirely and emits one generic body (`:130-132`). `fcmTokens` is written by nothing (`recipients.ts:23-29`) — sends are emulator-proven against `FakeMessagingPort`; device delivery rides operator item 4.
- **The settings screen is the designated home** (ADR-018 D7, `settings_screen.dart:26-38`): "KVKK export/delete rows land HERE and EXTEND this screen." `showDialog` is legitimate there (behind the Navigator, unlike the lock screen).
- **The sign-out path the deletion must land on** (ADR-018 / scout): the root listener `app.dart:89-96` wipes the lock on `next is AuthSignedOut`; `AuthSignedOut` is value-equal, so only a transition from a value-inequal state (here: `AuthSignedIn`) notifies. `AuthController.signOut()` (`auth_controller.dart:85-101`) is the shape to mirror: manual-op gate, explicit `state = const AuthSignedOut()` on success, `AuthError` on throw (protection stays). The lock screen's `_confirmRecovery` (`lock_screen.dart:197-221`) is the sign-out-first, settled-state-read, idempotent-wipe prior art.
- **The users update rule has no `hasOnly`** (`firestore.rules:33-35`) — a client can today write arbitrary NEW fields on its own doc; only `createdAt`/`coupleId` are frozen. Any field this ADR wants to be server-truth must gain its own freeze clause + mutation test (Decision 6, Decision 3).
- **Test machinery:** in-process service suites seed answer docs on the trigger-isolated `demo-hayati-notrigger` project (`functions/test/support/admin.ts`); rules mutations are in-memory string surgery with anchor-rot guards (`firestore-rules.test.ts:931-1660`); `FakeMessagingPort` byte-compares sends against `composePush` output. Coverage gates: functions 80 hard / 85 target, app 68.

**Decision clusters that came due (the ADR-013 scaffold):**
1. What "delete everything" MEANS for the two-person shared object (Decision 1).
2. The cascade mechanism: ordering, idempotency, resumability, the mid-cascade race surface (Decision 2).
3. The partner-notification contract: channel, content, B's morning-after state (Decision 3).
4. Entitlement handling — what "handled" means at the mirror level (Decision 4).
5. Export: scope, format, delivery (Decision 5).
6. The discreet-notification override — field, write path, resolver (Decision 6).
7. The app half: flow, confirmation honesty, the lock invariants (Decision 7).
8. Fail directions (Decision 8).

**Alternatives weighed at the top level:**
- *Author-split deletion (delete only A's authored halves; tombstone the couple; B keeps B's own answers).* Rejected — Decision 1 records the full argument and the honest costs of both options.
- *Soft delete with a grace window (7–30 days).* Rejected: `architecture.md:110` prescribes hard delete; a "deleted" state that is quietly recoverable is a guarantee-vs-mechanism inversion (the app would SAY deleted while the data sits restorable); and in the DV frame a restore path is a weapon — a victim who deletes to escape must not leave a lever that the person holding their phone, SIM, or Apple ID can pull to bring the thread back. Accidental-tap protection is carried by the two-step confirmation + PIN re-verify instead (Decision 7). KVKK's "without undue delay" does not require a cooling-off window.
- *Email delivery for export.* Rejected: needs an email provider = a NEW operator dependency the resume-prompt tells us to avoid; and an emailed JSON is a plaintext copy of relationship data sitting in an inbox the partner may share. In-app delivery is both cheaper and more protective (Decision 5).
- *Blocking deletion on partner consent.* Rejected without hesitation: a consent gate hands an abusive partner a veto over the victim's erasure right. Notification, never consent (`architecture.md:110` already says exactly this).
- *`beforeUserDeleted` blocking Auth trigger + client-side `currentUser.delete()`.* Rejected: blocking Auth triggers require the Identity Platform upgrade; and client-driven auth deletion hits `requires-recent-login`, putting a re-auth wall in front of a legal right. A callable that does everything server-side (auth deletion last) needs neither.

## Decision 1 — "Delete everything" means the couple is terminal: the shared thread dies with the couple, both halves; each partner's solo space is theirs alone

- **What A's deletion removes, exactly:** A's `users/{A}` + all `soloAnswers`; A's Firebase Auth record; every invite naming A (`creatorUid == A` or `joinerUid == A`); **the entire couple subtree** — `couples/{cid}`, every `days/{dayKey}`, every `answers/{authorUid}` doc **for both authors**, `subscriptions/{cid}` (Decision 4), `coachUsage/{cid}` and both `daily/{uid}` lanes; and the transient `deletions/{A}` progress doc (Decision 2). What survives: B's `users/{B}` doc (minus the `coupleId` pointer, plus the `coupleEnded` marker — Decision 3), B's `soloAnswers`, B's Auth record.
- **Why the whole thread and not just A's half.** Three arguments, in decreasing order of weight:
  1. **B's answers in the shared thread are mixed personal data.** They were written *to A, about the relationship with A*, inside a mutual-reveal structure; they routinely describe A. An erasure right that leaves the other side of every conversation readable by the partner has not erased the relationship record — it has redacted half of it. In the DV frame this is decisive: a victim who deletes to escape must not leave the abuser a curated archive of the relationship.
  2. **The governing docs already say it.** The accept line is *"couple data gone"* (`implementation-plan.md:69`) and `architecture.md:110` says *"cascade couple data"* — not "cascade the requester's half."
  3. **A half-thread is dishonest data.** B's answers were revealed *in exchange for* A's; stripped of their counterparts they misrepresent what B agreed to share and with whom.
- **The alternative, recorded loudly (the founder can overturn this with a superseding ADR):** *author-split deletion* — delete only `answers/{A}`, `daily/{A}`, `lanes[A]`; keep a tombstoned couple (`memberUids` scrubbed to `[B]` or a status flag) so B retains read access to B's own authored answers. Honest costs of the split: B keeps content that is substantially *about A* after A asked for erasure; the tombstone couple is a permanent ghost object needing new rules, new states, and a re-pairing story; and the app must explain to B why half of every day is missing — a worse product experience than a clean, honest ending. Honest cost of the chosen option: **B loses B's own authored couple-thread content without B's consent.** That is real, it is the price of an erasure right over a two-person object, and we say it to B plainly in the notification copy (Decision 3) rather than hiding it. B's solo reflections — the space that was only ever B's — are untouched.
- **What no mechanism can carry (stated so nobody claims it):** deletion removes the server copies. It cannot un-reveal what B already read, screenshot, or remembers. The confirmation copy must not promise otherwise (Decision 7 pins the copy to "deleted from Hayati", never "your partner will no longer have seen it").
- **Backups/PITR residual:** no deploy has ever happened and no PITR/backup schedule exists today, so "deleted" is currently literal. When the deploy era configures operational backups, backup-cycle retention must be aligned and documented (KVKK tolerates documented backup-cycle deletion delays). Recorded here so the first deploy session inherits it; not buildable now.
- **Deletion is per-user, not per-couple, on the requester's side:** an unpaired A (no `coupleId`) gets the same flow minus the couple steps — profile, solo answers, invites, Auth record. Test-pinned.

## Decision 2 — One idempotent, resumable cascade callable: `deleteAccount`, detach-first ordering, auth deletion last, every step re-drivable

- **The Function:** `deleteAccount` — `onCall`, `europe-west1`, `enforceAppCheck: false` (matches every callable), auth-guarded. The request must carry `confirm: 'DELETE'` (a wire-level literal; anything else → `invalid-argument`) so no client bug can invoke deletion by accident. Response: `{ status: 'deleted' }`. Errors follow the coach taxonomy: static messages, typed `details.reason`, **no content and no uids in any log line** (ADR-016 D8 / `architecture.md:110` discipline — deletion events are special-category-adjacent: log step names and counts, never identifiers beyond what Functions logging already stamps).
- **Why a progress doc exists at all:** after the detach transaction clears `users/{A}.coupleId`, a re-driven cascade could no longer FIND the couple subtree it still has to sweep. `deletions/{uid}` `{ coupleId, partnerUid, notifiedAtMs, startedAt }` is written **before** any destructive step and is the resume cursor. It lives under the rules catch-all deny (no client access; header-commented in `firestore.rules`). It is itself personal-data residue, so removing it is a cascade step, not an afterthought.
- **The pinned order (each step idempotent; re-drive = re-run from the top, completed steps no-op):**
  1. **Resolve + record.** Read `users/{A}.coupleId`; if paired, read `couples/{cid}.memberUids` → partnerUid. Merge-write `deletions/{A}`. If `users/{A}` is already gone (prior run died late), fall through with `deletions/{A}`'s recorded values, or — if that is gone too — skip to step 7.
  2. **The detach transaction** (single Firestore txn): delete `couples/{cid}`; on `users/{B}`: delete `coupleId` + set `coupleEnded: { at: serverTimestamp }`; on `users/{A}`: delete `coupleId`. **Guard: B is touched only if `users/{B}.coupleId == cid`** — on a re-drive after B has already re-paired, B's new couple must not be dented (test-pinned). Deleting the couple doc FIRST is the ordering lever: it instantly fail-closes both members' couple-scoped read/write surface (rules `get()` on the couple now denies everything — the mid-cascade race answer), while B's honest re-route rides the atomic `coupleId` clear, never a phantom-couple `permission-denied`.
  3. **Partner notification send** (Decision 3), latched by `notifiedAtMs` on `deletions/{A}` so a re-drive does not double-push. Best-effort by standing policy: absent tokens = loud counted skip; failure never aborts the cascade (the server-truth `coupleEnded` field from step 2 carries the guarantee).
  4. **Couple-subtree sweep** (admin list-and-delete; orphaned subcollections remain listable after the parent doc's deletion): every `days/{dayKey}/answers/*`, every `days/{dayKey}`, `coachUsage/{cid}/daily/*`, `coachUsage/{cid}`, `subscriptions/{cid}`. List-and-delete is naturally idempotent — a re-drive sweeps whatever remains.
  5. **Invites sweep:** query `invites` on `creatorUid == A`, then `joinerUid == A`; delete all hits (consumed invites' audit value yields to the erasure mandate; the couple they document is gone).
  6. **Own-subtree sweep:** every `users/{A}/soloAnswers/*`, then `users/{A}` itself.
  7. **Remove the cursor:** delete `deletions/{A}`.
  8. **Delete the Auth user** (`admin.auth().deleteUser(A)`) — **deliberately last**, so any earlier failure leaves A authenticated and able to re-drive the callable. If step 8 itself fails, the callable errors honestly; a retry re-enters at step 1, finds nothing left in Firestore, and re-attempts only step 8.
- **Kill-mid-cascade proof (the firm deliverable):** the service takes an injectable `checkpoint(step)` hook (the `deps.beforeWrite` prior art from `reveal-service.ts:47-58`); the test loop kills the cascade after each step k, re-drives, and asserts the full postcondition set converges. Concurrent double-delete (both partners simultaneously) converges by the same idempotency.
- **The token-window residual (recorded, not fixable at this layer):** Firestore rules validate the caller's ID token, and a deleted user's ID token stays valid for up to ~1 hour (rules have no revocation check). Within that window a hostile client could re-write `soloAnswers` or re-create `users/{A}`. Mitigations shipped: the couple surface is already fail-closed (step 2); `soloAnswers` writes gain an `exists(users/$(uid))` guard in rules (mutation-tested — Decision 6 carries the rules diff), so solo orphan-writes die with the profile doc. The residual that remains: a re-created `users/{A}` profile doc within the token hour — an unreadable-forever orphan (the uid never signs in again), equivalent in effect to a fresh sign-up, requiring a deliberately hostile client. Accepted and named; not silently absorbed.
- **New-writer governance:** this Function is a new admin-SDK writer over collections whose doc-comments already anticipate exactly it. `revenueCatWebhook` remains the sole writer of `subscriptions/{cid}` **content**; the cascade only ever deletes that doc whole (never edits lanes) — the ADR-013 sole-writer claim narrows to "sole content writer," recorded in the §3 docs pass.

## Decision 3 — Partner notification: a server-truth field written atomically with the detach, plus a deliberately name-free push; B's morning-after is an honest terminal screen

- **The load-bearing channel is Firestore, not FCM.** `users/{B}.coupleEnded = { at: Timestamp }` is written in the detach transaction (Decision 2 step 2) — atomic with the moment B's couple ceases to exist, present on every device B signs into, and deliverable today (unlike push, whose device half waits on operator item 4). It carries **no reason, no message, and no reference to A** — B's client needs none of those to render the honest state, and A's identifiers on B's doc would be residue the cascade just promised to remove.
- **Field governance:** `coupleEnded` is server-truth: the users update rule gains a freeze clause (same shape as the `coupleId` freeze; clients can neither mint nor clear it), mutation-tested. It is cleared by the next successful `joinInvite` (the join transaction deletes it from both members — a one-line extension), so a re-pairing B does not carry a stale tombstone. B's "seen" state is device-local (`LocalFlagStore`, the ADR-017 seam), so the notice shows once per device without a server ack write.
- **The push:** a fourth `PushKind`, `'coupleEnded'`, through the existing `composePush`/`MessagingPort` mold. **Its normal-mode copy is deliberately name-free and reason-free by design** — unlike `partnerAnswered`, there is no named variant at all: "your partner ended your shared space" on a lock screen is a relationship-status disclosure to anyone holding the phone, and the person most likely to be looking is exactly who the DV posture worries about. Copy (TR/AR/EN, drafted in the brandkit voice, native review flagged): title `Hayati`, body ≈ "Your shared space in Hayati has been closed. Open the app when you're ready." Discreet mode falls through to the standard generic `DISCREET_BODY` untouched (`payload-policy.ts:130-132` needs no new code — by construction). Quiet hours suppress (drop-with-loud-log, the standing M3.4 policy); the field from step 2 is why suppression costs nothing. The ADR-012 absolute invariant holds by construction: `composePush` still has no question/answer parameter.
- **B's morning-after state (the honest terminal screen):** the onboarding gate today routes `coupleId == null` → solo home silently (`onboarding_gate.dart:59-66`) — which would make B's couple *vanish wordlessly*, a dishonest state. The gate gains one branch: `coupleId == null && coupleEnded != null && !seenLocally` → `CoupleEndedNoticeScreen`. Its copy (×3 locales, native review flagged, DV-neutral, non-blaming): the shared space was closed by your partner and its content — both sides of it — permanently deleted; your own private reflections are untouched and remain yours; if you had a subscription, it is not cancelled by this — manage it in your App Store settings; you can pair again whenever you choose. One primary action ("Continue") → sets the device-local seen flag → solo home. If B is ON the paired home at the deletion instant, the profile stream (which the gate already watches) emits the cleared `coupleId` and re-routes live — usually before the orphaned couple listener can even error; the existing `_PairedErrorView` remains the fallback for the losing side of that race, and the notice screen renders on the next gate evaluation.
- **What "partner notified" honestly means this session:** the server field is guaranteed and atomic; the push logic is emulator-proven against `FakeMessagingPort` (byte-compared to `composePush` output); zero real devices receive anything until `fcmTokens` capture + APNs ship (operator item 4) — the same posture every M3.4 notification already holds. Stated here so the accept line is read honestly.

## Decision 4 — Entitlement "handled" = the mirror dies with the couple (ADR-013 Option A); the real subscription is out of reach and we say so

- **The cascade deletes `subscriptions/{cid}` whole** in the subtree sweep. "Absent doc = free tier" is already the defined read model (ADR-013 D5), so this is a well-defined downgrade, not an undefined state. Both members read free immediately (B via the honest notice; B's couple is gone anyway — there is no surviving couple context to be premium).
- **Alternatives rejected:** *lane-scrub + re-derive* (ADR-015-consistent only if a single-member couple survives, which Decision 1 rejected); *tombstone the payer's lane* (couples the cascade to the webhook's LWW order-key model for zero benefit once the doc is deleted anyway); *do nothing* (leaves an orphan uid-bearing doc — data residue in a flow whose purpose is erasure — plus a paid-but-downgraded freeze window).
- **What this deliberately does NOT claim:** the cascade cannot cancel, refund, or even see the real RevenueCat/App Store subscription — those are RC *account* operations behind the secret key that does not exist until operator item 0 (the stopping condition in the resume-prompt anticipated exactly this boundary; the RC-REST reconciliation debt already rides item 0 + the deploy era). Consequences, named per payer:
  - **A (the deleter) was the payer:** Apple keeps billing A's Apple ID until A cancels in App Store settings. A's confirmation screen says so in plain words (Decision 7) — deletion of the account is not cancellation of the subscription, and pretending otherwise would be the exact over-claim class S020 taught us to delete.
  - **B (the survivor) was the payer:** B pays for a couple that no longer exists. B's notice screen carries the same static App-Store-settings line (static for everyone — the copy never reveals *who* paid, so it leaks nothing). If B re-pairs, the next RC lifecycle event resolves through `users/{B}.coupleId` and mints the mirror on the new couple (`resolveCouple` mechanism, no new code) — stated as mechanism, not promised as UX (an annual subscription's next event can be months away; the reconciliation debt covers the gap when it lands).
- **Webhook aftermath:** post-cascade RC events for A resolve to nothing → `unresolvable` → loud 200 skip (`entitlement-service.ts:234-241`) — already the designed behavior, no code change, no error noise.

## Decision 5 — Export: one callable returning a versioned JSON document, delivered in-app; strictly the requester's own data; no premium gate; no email provider

- **The Function:** `exportData` — `onCall`, auth-guarded, **no premium gate and no cap** (a data-subject right is not a feature tier; personal-use scale makes abuse a deploy-era concern, recorded with the existing rate-limiter hardening note). Returns the export document directly in the callable response.
- **Scope — everything A authored or that is about A, nothing the partner did (the firm deliverable's words):**
  - `profile`: `users/{A}` client fields (`status`, `contentLanguage`, `register`, `createdAt`) + the Auth record (`displayName`, `email`, `photoURL`) + `notificationPrivacy` if set.
  - `soloAnswers[]`: dayKey, questionId, question text (app content, included for readability), answer text, answeredAt.
  - `coupleContext` (when paired): `coupleId`, `createdAt`, `timezone`, `streak` (jointly derived, member-readable by design), the coach monthly counter (member-readable by design). **`memberUids` and every other appearance of B's uid are excluded** — the partner's identifier is the partner's data.
  - `coupleAnswers[]`: **only `answers/{A}`** — dayKey, questionId, question text, A's answer text, answeredAt, whether that day revealed. Never B's text, revealed or not — mirroring the export to the reveal gate would still hand A a fresh copy of B's words at export time; we simply never include them. (This also closes the pre-reveal side channel `architecture.md` §4 worries about, by construction.)
  - `coachUsage`: A's own `daily/{A}` lanes only — the ADR-016 D7 self-read invariant applied to export verbatim; B's usage frequency never crosses.
  - `subscription`: the summary + `lanes[A]` only.
  - `invites[]`: codes A created or redeemed — code, status, timestamps; **the counterpart uid scrubbed**.
  - Envelope: `{ formatVersion: 1, generatedAt, uid, data: {…} }`. Field names are a wire contract (the profile enum-`name` precedent, `architecture.md:72`); `formatVersion` exists so a future shape change is explicit.
- **Delivery: in-app.** The app renders the document on an export screen (selectable, copy-to-clipboard). No email provider, **no new operator item** — this is the deliberate reversal of `architecture.md:110`'s "email" parenthetical (see Supersedes note; the resume-prompt licenses exactly this preference). A share-sheet/file export is a recorded future nicety, not v1.
- **Size honesty:** at this product's shape (≤2000-char answers, one per person per day), years of use fit in tens-to-hundreds of KB against a ~10MB callable ceiling. The bound is real but distant; if it is ever approached, the recorded upgrade path is pagination or a signed Storage URL — a new ADR, not a silent stretch.
- **What export deliberately does NOT include:** B's anything (stated above); coach conversation content (none exists — ADR-017 ephemeral; if operator item 7 ever persists `coach_sessions`, its self-read-only bundle explicitly includes M6 export/cascade integration — recorded, not silently omitted); the device-local Keychain lock record (device data, wiped by sign-out per ADR-018, never server-resident); crisis/help-path linkage (ADR-016 D8 — no such join exists server-side and the export must never reconstruct one).

## Decision 6 — The discreet-notification override: `users/{uid}.notificationPrivacy`, written by a settings Function, read through the `resolveDiscreet` seam; rules freeze + the soloAnswers existence guard are this session's only rules changes

- **The field:** `users/{uid}.notificationPrivacy: 'discreet'` — a string enum with exactly one legal value in v1, absent meaning "locale default". An enum rather than a boolean so a future `'named'` value (an AR-locale user opting OUT of the discreet default) is representable without migration. **v1 is deliberately opt-in-only**: the AR default (PRD F6) stays non-overridable — weakening a protective default is a separate product decision, recorded here, not smuggled in.
- **The write path:** callable `updateNotificationPrivacy({ discreet: bool })` — sets the field or deletes it. Server-validated; the users-doc client-write surface stays exactly as it was. This follows the M6.1 deferral's own words ("a `users` field + a settings Function") — and it is made TRUE by a rules change: the users update rule gains a `notificationPrivacy` freeze clause (the `coupleId` freeze shape), so "server-owned field" is a mechanism, not a sentence. Without the freeze, the claim would outrun the rules (the users update rule has no `hasOnly` — scout finding), which is precisely the S020 finding class.
- **The read path:** `resolveDiscreet(contentLanguage, notificationPrivacy)` → `notificationPrivacy === 'discreet' || contentLanguage === 'ar'`. Junk values are ignored (the `contentLanguageOf` defensive-reader pattern). Both call sites pass the field from the `userData` they already hold — the one-function-plus-two-args change the seam comment promised. `coupleEnded` pushes resolve discreetness identically (B's own field/locale).
- **The settings row:** a `SwitchListTile` on the settings screen — value = the explicit field (streamed via the existing profile watch; `ProfileDto` maps the new field), busy/error per the `_ErrorLine` idiom. For AR-locale users the subtitle states the default is already on; flipping the switch on writes the explicit value anyway (deliberately: an AR user who later changes `contentLanguage` keeps their discreet posture instead of silently losing it — the honest reason the explicit write matters).
- **This session's complete rules diff (both mutation-tested, nothing else moves):**

  2. `soloAnswers` create/update: add `exists(/databases/$(database)/documents/users/$(uid))` — the deleted-user orphan-write guard (Decision 2). Positive control: a normal write with a live profile still succeeds; mutation: weaken → the orphan write succeeds.

New delete-deny mutants for `users`/`soloAnswers`/`couples` (the three the harness notes as uncovered) are added as test-coverage, not rules changes.

## Decision 7 — The app half: two rows + a toggle on the existing settings screen; deletion lands on `AuthSignedOut` from signed-in; the four lock invariants untouched

- **Surface:** the M6.1 `SettingsScreen` gains "Download my data", "Delete account & data", and the Decision-6 toggle — extending the existing `ListView` (ADR-018 D7's explicit reservation; no second surface). Everything stays behind the lock like the rest of the app — deliberately (resume-prompt handoff): a pre-auth/pre-unlock deletion entry point would be its own ADR.
- **The export flow:** row → `ExportScreen` (pushed, the `PinSetupScreen` mold) → calls the repository → renders the JSON with copy-to-clipboard; loading and honest error states per the coach taxonomy mapping. A "Download your data first?" link sits inside the deletion flow, one tap away.
- **The deletion flow — never a one-tap accident, and honest to the letter:**
  1. Row → `DeleteAccountScreen` (pushed) stating, in plain sentences ×3 locales (native review flagged): this is **irreversible**; it deletes your account, your solo reflections, and the **entire shared space — both sides of every answer** — permanently; your partner will be notified that the space was closed (not why); **it does not cancel any App Store subscription** — manage that in your App Store settings; consider downloading your data first. The copy never claims the partner "loses what they've already seen" (Decision 1's honesty bound).
  2. Confirm → if the lock is enabled, `_PinVerifyDialog` (the existing seam — attempt-bounded, cooldown-aware; any biometric anywhere stays behind the `biometricOnly` sentinel, untouched); if no lock, a destructive-styled confirmation dialog naming the word "permanently". `showDialog` is legitimate here — settings is behind the Navigator (Invariant D applies only to lock-screen children).
  3. `AuthController.deleteAccount()` — shaped exactly like `signOut()` (`auth_controller.dart:85-101`): the manual-op gate spans the whole operation (so a mid-cascade spontaneous stream `null` — the server deletes the Auth user — cannot clobber the in-flight state); it calls the data-rights repository's `deleteAccount` callable, then local `signOut()` (session teardown; the server already deleted the Auth record), then sets `state = const AuthSignedOut()` explicitly. The pre-state is `AuthSignedIn`, so the transition is value-inequal and the root listener (`app.dart:89-96`) fires the lock `wipe()` — the resume-prompt's binding handoff, satisfied on the existing path. Any throw → `AuthError`: **protection stays**, the screen shows honest retry copy, and re-driving is safe by Decision 2.
  4. Belt-and-suspenders (the `_confirmRecovery` prior art): after the op the delete screen reads the **settled** auth state and calls `wipe()` directly if `AuthSignedOut` — idempotent, and immune to the value-equality trap by construction.
  - **Invariant audit:** (A) nothing invalidates `privacyLockControllerProvider` — the wipe is reached via the root listener + direct `wipe()`; (B) no new mutating method is added to the lock controller at all; (C) `biometricOnly` untouched — deletion re-auth is PIN-only through the existing dialog; (D) no dialogs above the Navigator — all new UI is pushed routes/dialogs below the gate.
- **Free tier untouched, assertion-proven:** the two rows + toggle render for free users (a data right is not premium); the daily loop and every pre-existing golden stay byte-identical (the M5.2 probe-pin precedent).

## Decision 8 — Fail-direction rules (the table reviewers check first)

| Failure / vector | Direction | Why |
|---|---|---|
| Cascade dies mid-run (any step) | **Resumable** — callable errors honestly; A stays authenticated (auth delete is last); re-drive converges (idempotent steps + `deletions/{uid}` cursor) | never a permanently half-deleted couple (D2) |
| Re-drive after B re-paired | B's new couple **untouched** — detach guard `users/{B}.coupleId == cid` | a stale cascade must not dent a new relationship (D2) |
| Both partners delete concurrently | **Converges** — idempotent deletes, per-uid cursors | (D2) |
| Push send fails / no tokens / quiet hours | **Dropped, counted, loud** — the atomic `coupleEnded` field carries the guarantee | best-effort push is the standing M3.4 policy (D3) |
| Server delete OK, local sign-out throws | `AuthError`; **lock STAYS**, honest retry | protection never drops on an error path (D7, ADR-018 asymmetry) |
| Deleted user's still-valid token writes | Couple surface **fail-closed** at detach; soloAnswers **fail-closed** by the exists-guard; users re-create = recorded residual (≤1h, hostile client, unreadable orphan) | (D2, D6) |
| RC event for the deleted payer arrives | `unresolvable` → loud 200 skip, no write, no error noise | existing ADR-013 D3 behavior (D4) |
| Payer's real subscription | **Out of reach — billing continues until cancelled in App Store**; said plainly in both users' copy | RC account state is behind operator item 0 (D4) |
| Export larger than the callable ceiling | Callable errors honestly; recorded upgrade path (pagination / signed URL) | distant bound, named not silent (D5) |
| B on the paired home at deletion instant | Profile stream re-routes to the notice; `_PairedErrorView` is the race fallback | no crash, no ghost couple (D3) |
| `coupleEnded` faked by a client | **Impossible** — rules freeze, mutation-tested | server-truth field (D3, D6) |
| Backups (deploy era) | Recorded obligation: align backup retention with erasure; nothing exists today | (D1) |

## Consequences

**Positive**
- The last legally-required MVP feature ships emulator-proven; the M6 accept line's three clauses (gone / notified / handled) each have a pinned mechanical meaning and a test.
- The cascade honors every promise the rules have carried since M2 without loosening a single client rule — deny-by-default survives the feature that most tempts violating it.
- The deleting user gets a total, immediate, unvetoable exit (the DV-decisive property); the surviving partner gets an atomic, honest, non-blaming account of what happened and keeps everything that was ever theirs alone.
- No new operator dependency: export is in-app; the notification's guaranteed channel is Firestore; the push half rides existing seams.
- The `resolveDiscreet` seam is finally filled exactly as its comment promised — one function, two args, one new callable, one freeze clause.
- The wire surface (`deleteAccount` confirm literal + error taxonomy, `exportData` envelope `formatVersion: 1`, `updateNotificationPrivacy`) is **frozen for the app half of this session and anything later**.

**Negative (accepted trade-offs)**
- **B loses B's own authored couple-thread content without consent** — the honest price of hard erasure over a two-person object; said to B plainly, alternative recorded for the founder (D1).
- **Billing does not stop at deletion** — the payer (either one) must cancel in App Store settings; the mirror cannot reach the subscription (D4; RC-REST debt rides item 0).
- **The ≤1h token-window residual** (hostile-client orphan re-writes; solo surface guarded, users re-create accepted and named) (D2).
- **Push delivery is still logic-only** until operator item 4 (APNs + token capture) — the server field carries the guarantee meanwhile (D3).
- **No grace window** — an accidental deletion (two confirmations + PIN deep) is unrecoverable; that is what "hard delete" means and the copy says so (D1, D7).
- Deploy-era obligations recorded, not built: backup-retention alignment, export rate-limiting, RC reconciliation.

**Neutral**
- Coach content stays out of scope because nothing persists (ADR-017); the `coach_sessions` bundle (operator item 7) explicitly includes export/cascade integration when the founder decides.
- The AR discreet default remains non-overridable in v1; the enum leaves the door open without migration (D6).
- `deletions/{uid}` is a transient, client-invisible collection; its removal is itself a cascade step.
