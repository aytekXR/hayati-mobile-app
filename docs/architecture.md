# Architecture — Hayati

## 1. Stack & rationale

| Layer | Choice | Why |
|---|---|---|
| App | **Flutter** (stable channel), Dart | One codebase for Android-heavy TR + iOS-heavy GCC; first-class RTL; widget-test culture fits our TDD mandate |
| State | Riverpod (+ codegen) | Testable, compile-safe DI, scales from MVP to v2 |
| Backend | **Firebase**: Auth (Google/Apple/phone), Cloud Firestore, Cloud Functions (TypeScript), FCM, Remote Config, Crashlytics, App Check | Solo-founder ops budget ≈ zero; offline persistence built-in; Remote Config gives the regional feature flags the GCC strategy requires |
| Payments | **RevenueCat** | Cross-store entitlements, webhooks, experiments; industry default for this exact app category |
| Analytics | Mixpanel (product funnels) + Firebase Analytics (audiences) | Free tiers cover Year 1; funnel tooling for the three gates |
| AI | LLM via **Cloud Function proxy only** | API keys server-side; per-user rate limits; safety filter; model-swappable; cost caps enforceable |
| CI/CD | GitHub Actions + Fastlane | Matches `project-rules.md` (gh CLI, pipeline-green rule) |

Firestore region: **europe-west (multi-region EU)** — defensible latency for both Istanbul and Riyadh, and a conservative posture for KVKK/PDPL cross-border rules. Revisit if KSA data-residency guidance tightens; the repository pattern below keeps a migration survivable.

## 2. Repository & module layout

```
hayati/
  app/                         # Flutter
    lib/
      core/                    # design_system/ l10n/ analytics/ storage/ config/ utils/
      features/
        auth/  pairing/  daily_question/  streak/
        thread/  coach/  paywall/  privacy_lock/  settings/
        # each feature: domain/ (entities, usecases) data/ (repos, dtos) presentation/ (state, widgets)
      app.dart  main_dev.dart  main_prod.dart      # flavors
    test/          # mirrors lib/, unit + widget
    integration_test/
  functions/                   # TS Cloud Functions: coach_proxy, pairing, rc_webhooks, question_rollover, cleanup
  content/                     # question packs as versioned JSON (tr_playful, tr_respectful, ar_msa_gulf, en) + validator script
  docs/                        # this documentation set (source of truth)
  .github/workflows/           # ci.yml, release.yml
  fastlane/
```

Feature-first, clean-ish architecture: `presentation → domain ← data`. Domain layer is pure Dart (no Flutter imports) — this is what makes the 90% domain coverage target in `test-suite.md` cheap.

**Firebase environments (M1.2):** two real Firebase projects map to the two Dart-entrypoint flavors — **`hayatiapp-dev`** / **`hayatiapp-prod`** (`com.hayati.app` on both platforms; the planned `hayati-dev`/`hayati-prod` ids were unavailable). Options live as committed per-flavor Dart files (`app/lib/core/firebase/firebase_options_{dev,prod}.dart`, generated by `flutterfire configure`, FlutterFire Dart-only init — no `GoogleService-Info.plist`/`google-services.json` committed and no google-services Gradle plugin, deliberately: one Runner/one Android app serves both flavors, so platform config files can't be per-flavor; config files are not secrets, API keys get console-side restrictions). Google sign-in OAuth client ids are likewise per-flavor **runtime** config (`app/lib/core/firebase/google_sign_in_config.dart`: iOS `clientId` ≙ `GIDClientID`, Android web `serverClientId`), with both flavors' iOS `REVERSED_CLIENT_ID` URL schemes registered side by side in `Info.plist`. *Provisioning learnings (Session 004):* free-tier Firebase **Auth initialization and Google-provider enablement are console-only** — the Identity Toolkit Admin API rejects provider creation without an existing OAuth client and `identityPlatform:initializeAuth` is the billing-gated Identity Platform upgrade; everything else (API enablement, `flutterfire configure`, Firestore database creation at `eur3`, rules deploy) is scripted via `firebase-tools`/REST. Environment bindings (Firebase-backed `AuthRepository`, Firestore-backed `ProfileRepository`) are composed in the entrypoints via `runHayati(..., extraOverrides: [...])`; base providers throw when unoverridden (the `appConfigProvider` pattern). Emulators stay opt-in dart-defines (`USE_AUTH_EMULATOR`, `USE_FIRESTORE_EMULATOR`, and since M2.2 `USE_FUNCTIONS_EMULATOR` — wiring the `europe-west1` `FirebaseFunctions` instance, the same regional instance the pairing repository resolves; the Firestore emulator needs **Java 21+**, relevant to ci-debt #6); runtime verification path is `app/integration_test/`. Firebase iOS SDK 12 raised the iOS floor: deployment target pinned to **15.0** in the Runner pbxproj (iOS plugins integrate via **Swift Package Manager** — the Flutter 3.44 scaffold is SwiftPM-first and has no Podfile; adding one breaks the build, verified on CI in Session 003).

**Auth providers (M1.3).** Three providers reach Firebase through one uniform seam in `features/auth/data/`: a gateway per provider that yields a Firebase `AuthCredential` (or `null` when the user cancels), which `FirebaseAuthRepository` exchanges via `signInWithCredential` inside its `_guarded` boundary. This shape exists because `signInWithCredential` forwards the credential unvalidated, so the Auth emulator accepts an *unsigned* JSON `id_token` — which is what makes every provider testable without a real account ([ADR-008](adr/008-apple-signin-via-credential-seam.md)). **Apple** uses `sign_in_with_apple` + `AppleAuthProvider.credentialWithIDToken` (never `signInWithProvider`, which surfaces no credential and therefore cannot be faked or emulated); the nonce is a two-value protocol — SHA-256 hex to Apple, plain `rawNonce` to Firebase. **Phone** is the one flow with a user-input gap, so it cannot be a single atomic call: `PhoneAuthGateway` adapts `verifyPhoneNumber`'s four callbacks into one `Completer` (guarded against double-completion — `verificationFailed`/`codeAutoRetrievalTimeout` can fire *after* `codeSent`, and the returned Future can reject independently), and a screen-scoped `PhoneSignInController` drives enter-number → code-sent → confirm. That controller **never writes the global `AuthState`**: the terminal `AuthSignedIn` arrives on the `authStateChanges` stream while `AuthController` is idle, preserving its manual-op precedence contract. On iOS `verificationCompleted` never fires (Android-only), so a code-entry UI is mandatory; on Android it fires *instead of* `codeSent` under instant verification, which would hang the send forever — the gateway therefore fails loudly there until real Android support lands (issue #13, M6.5). Emulator-side, phone codes are minted randomly per request and read back over the emulator's REST endpoint — **under the project the emulator was booted with (`demo-hayati`), not the app's `FirebaseOptions.projectId`** (the Auth emulator resolves every SDK request to its single `--project`; querying `hayatiapp-dev` returns an empty list, verified 2026-07-09).

**Crashlytics & App Check (M1.3).** Both are activated **only from the flavor entrypoints**, after `initializeFirebase` — never inside it, and never from `HayatiApp` — because `FirebaseAppCheck.activate()` and `FirebaseCrashlytics.instance` each drive a platform channel that throws in the plain `flutter test` VM, where the bootstrap and widget trees are unit-tested. Crash reporting crosses into the app behind a `CrashReporter` interface taking only framework types (`core/observability/`); exactly one file imports `firebase_crashlytics`, and `runHayati(..., crashReporter:)` installs `FlutterError.onError` + `PlatformDispatcher.onError` hooks only when a reporter is supplied, so tests stay channel-free. **Per-flavor collection policy: dev OFF (noise), prod ON**, applied at boot via the runtime `setCrashlyticsCollectionEnabled` API (it persists across launches and overrides plist defaults, so a device that switched flavors self-corrects). **App Check posture:** dev activates the *debug* provider, prod declares *App Attest* (always available at the min iOS 15.0 floor); activation is skipped entirely under `kUseAuthEmulator` so the integration leg stays hermetic. **Enforcement must stay OFF in both consoles until on-device attestation is verified** — enabling it early would lock the founder couple out of Auth/Firestore. Sign in with Apple adds `app/ios/Runner/Runner.entitlements` (`com.apple.developer.applesignin`), wired via `CODE_SIGN_ENTITLEMENTS` in the three Runner build configs; entitlements are consumed at codesign time, so CI's `--no-codesign` build is unaffected. *Deferred loudly to the on-device/Mac slice (nothing silent, rule #9):* App Attest entitlement + console app registration, per-device App Check debug-token registration, the `FirebaseCrashlyticsCollectionEnabled=false` Info.plist baseline that would close dev's first-launch pre-Dart collection window, dSYM upload for prod symbolication (the CocoaPods podspec adds it; SwiftPM does not), and real-device verification that Apple's first-authorization full name reaches `displayName`.

## 3. Data model (Firestore)

```
users/{uid}: status, contentLanguage, register, coupleId?, fcmTokens, createdAt
couples/{coupleId}: memberUids[2], timezone, packConfig, streak{count, lastMutualDate, graceTokens}, createdAt
  # shipped M2.3: memberUids [creatorUid, joinerUid] (creator first, frozen), timezone, createdAt; packConfig/streak arrive with M3
couples/{cid}/days/{yyyymmdd}: questionId, answers{uid: {text, at}}, revealedAt, thread[]
questions/{qid}: locale, register, category, depth, seasonalWindow?, text   # read-only, shipped+synced
invites/{code}: creatorUid, status ('pending'|'expired'|'joined'), expiresAt, createdAt, coupleId?, joinerUid?, joinedAt?   # M2.1: invite precedes the couple, so creatorUid; M2.3: join stamps the terminal 'joined' + coupleId/joinerUid/joinedAt (codes are never resurrected)
subscriptions/{coupleId}: entitlement mirror via RevenueCat webhook (source of truth = RC)
coach_sessions/{uid}/msgs: rolling window, TTL 30d
```

**Security rules invariants (tested against emulator):** a user reads/writes only their own couple's docs; partner's answer for today is unreadable until requester's answer exists (mutual-reveal enforced server-side, not just UI); invites expire; entitlement docs are function-write-only. *Shipped so far (M2.1 + M2.3, `firestore.rules`):* `users/{uid}` self-only with granular ops — `createdAt` **rules-enforced** create-once (`== request.time` at create, frozen on update — the M1.2 client transaction shape passes unchanged), client delete denied (hard delete is the M6 cascade Function), and since M2.3 **`coupleId` frozen against clients** (create may not include it, update may not set/change/delete it — `get('coupleId', null)` compared symmetrically so the app's merge writes pass; the field is authorization-meaningful, granting `couples/` access via `memberUids`, and is written only by the join Function); `couples/{coupleId}` member-only read/update with `memberUids` frozen (couple create/delete is exclusively the M2.3 transactional-join Function / M6 cascade); `invites/{code}` **no client access at all** — writes come from `createInvite`/`joinInvite` (admin SDK) and the invitee-facing read is the M2.2 zero-auth preview Function, so codes can't be enumerated client-side; default-deny for everything else. Every invariant is proven by `functions/test/emulator/firestore-rules.test.ts`, including **mutation tests** that load a weakened copy of the rules and show the denial disappears — the suite goes red if a protecting clause is commented out. Client wire mapping for `users/{uid}` lives in `app/lib/features/profile/data/profile_dto.dart` — enum `name`s are the wire values and renaming one is a data migration.

**Invite issuing (M2.1, `functions/`).** The repo's first Cloud Functions code: TypeScript strict on **firebase-functions v7 / Node 20** (firebase-admin pinned `^13` — v14 requires Node ≥22), vitest + v8 coverage with the `test-suite.md` §3 gate (target 85%, **hard fail <80%** via thresholds), eslint flat config. `createInvite` is an **onCall v2 callable in `europe-west1`** (closest single region to both Istanbul and Riyadh, consistent with eur3; the M2.2 client must construct its callable with this region): auth required (guarding `auth.uid` itself, not just presence — the functions emulator skips real token verification, and it's free defense-in-depth), App Check **token plumbed but enforcement OFF** (`enforceAppCheck: false` until on-device attestation is verified, M1.3 posture; token presence is logged to observe attestation coverage pre-enforcement). Codes are 8 chars from a 31-char unambiguous alphabet (A–Z + 2–9 minus O/I/L — survives being read aloud or retyped from WhatsApp; 31⁸ ≈ 8.5e11). **Re-issue policy: one active invite per creator** — a pending, unexpired invite is returned as-is (idempotent; re-opening the share flow never invalidates the code the partner already has), stale pendings are lazily marked `expired` and replaced; invites never accumulate. Everything runs in one Firestore transaction (pending-invite query + collision probes are reads-before-writes; `tx.create` fails if the code doc appeared meanwhile), so concurrent calls serialize and the loser returns the winner's code — proven against the emulator. Emulator wiring gotchas that cost real time, recorded once: the functions emulator **never compiles TS** (build `lib/` before `emulators:exec`), injects **no functions-host env var** into the exec child (tests derive `http://127.0.0.1:5001/<project>/<region>/<name>` from the pinned port), firebase-tools finds Java **via PATH only** (JAVA_HOME is never read; the firestore emulator needs Java 21+), and the functions emulator serves functions **only under its `--project` id** — a Flutter app initialized with a real flavor's `FirebaseOptions` builds the emulated callable URL from *its* `projectId` and 404s (`NOT FOUND`), because unlike the auth/firestore emulators (which resolve any project id) the functions emulator routes by URL path; emulator-backed integration tests must therefore initialize their default app with `demo-hayati` options (found the hard way on the first `integration-emulator` run of M2.2, fixed in PR #23).

**Invite preview (M2.2, `functions/`).** `invitePreview` is an **onRequest v2 HTTP endpoint in `europe-west1`**, zero-auth by design — the invitee has no account yet, so a callable's auth/App Check envelope would be dead weight (the onCall-vs-onRequest split was pre-decided in Session 007, not re-contested). Contract: `GET ?code=` → HTTP 200 `{status: 'valid'|'expired'|'unknown', creatorDisplayName?}`. **The response surface is exactly those fields** — never `creatorUid`, other invites, or (once M3 content exists) answer content; the suite's field-surface tests assert `Object.keys(body)` per case, so an accidentally added field turns CI red. This is where the §4 invariant "preview never leaks answer content" starts; the projection lives in one typed helper so the M3 growth (`questionText` + locked indicator) stays an explicit, reviewed surface change. Lookups return a **uniform 200/`{status}` shape for all three outcomes** (no enumeration oracle beyond validity — `expired` vs `unknown` only distinguishes a code you already hold), and malformed codes (checked against the alphabet/length imported from `invite-code.ts`) short-circuit to `unknown` without a Firestore read. The endpoint is strictly **read-only**: a stale pending is *reported* `expired` by comparing `expiresAt`, never written — a zero-auth path must not amplify writes; lazy expiry stays `createInvite`'s job (proven by a test that previews a stale invite and re-reads the doc). `creatorDisplayName` comes from the creator's **Auth record** (`users/{uid}` stores no display name) and is omitted — still `valid` — on any lookup failure. Logs avoid becoming an oracle: a 3-char code prefix is logged, never validity-plus-full-code. **Rate-limiting decision (recorded per resume-prompt):** a best-effort **in-memory fixed-window per-IP limiter (30 req/min → 429)** guards the endpoint — per-instance and reset on cold start, so it is an abuse damper, not a guarantee; the real anti-enumeration defense is the 31⁸ (~8.5e11) code space plus function-only reads. Infrastructure-grade limiting (Cloud Armor / API Gateway) is deliberately out of scope until a public launch surface exists. No CORS: the native app is the only consumer; a future web preview adds it deliberately.

**Transactional join (M2.3, `functions/`).** `joinInvite` is an **onCall v2 callable in `europe-west1`** (auth required with the same uid-not-just-presence guard as `createInvite`; App Check plumbed, enforcement OFF — M1.3 posture). Request `{code, timezone?}` → `{coupleId}`. Everything happens in **one Firestore transaction** (all reads before writes): read `invites/{code}` → both `users` docs, check in a deliberate order (existence → terminal-state → self-join → profile-missing → already-paired), then create `couples/{auto-id}` (`memberUids: [creatorUid, joinerUid]` — creator first, frozen), set `coupleId` on both `users` docs, and stamp the invite `{status: 'joined', coupleId, joinerUid, joinedAt}`. **Typed error surface (frozen wire contract the app maps on):** `not-found`/`{reason:'unknown'}` for absent or malformed codes (malformed short-circuits without a read — no enumeration oracle); `failed-precondition` with `reason` `'expired'` (status or clock or malformed doc), `'consumed'` (status `'joined'` — the race loser's answer), `'self-join'`, `'already-paired'` (either side has `users/{uid}.coupleId`), `'profile-missing'` (either `users` doc absent — the app flow guarantees profile-before-join; this is defense-in-depth). **Race semantics are the acceptance criterion:** two concurrent joins serialize on the invite doc; the admin SDK retries the ABORTED loser, whose retry re-reads the invite — now `'joined'` — and throws `consumed`; typed rejections always come from re-read state, never from inspecting commit conflicts (proven at service AND callable level against the emulator, exactly one couple per code). **Couple `timezone` source (documented decision):** the joiner MAY supply an IANA id, validated against `Intl.supportedValuesOf('timeZone')`; absent/unrecognized falls back to `Europe/Istanbul` (founder-couple / TR soft-launch home zone). The app currently sends nothing — a reliable device IANA source needs a plugin, deferred to M3 where the rollover work makes it load-bearing. `createInvite` gained the symmetric guard: an already-paired creator gets `failed-precondition`/`'already-paired'` instead of a fresh code, so re-issue after a join can never resurrect or replace a spent code (a `'joined'` invite also previews as the uniform `'expired'`).

## 4. Key flows

- **Pairing:** Function issues code/deep link → invitee hits preview endpoint (public, rate-limited, exposes question text + *locked* indicator only — never answer content) → on signup, transactional join, second-join races rejected. *Status:* creator side shipped M2.1/M2.2 — `createInvite` + the invite share screen (code + expiry + system share sheet with a WhatsApp-formatted TR/AR/EN message composed from ARB placeholders); preview endpoint shipped M2.2 with the §3 field surface (question text joins it at M3, when questions exist). **Joiner side shipped M2.3:** the partner preview screen consumes `pendingInviteProvider` (deep-link entry) with a manual code-entry fallback, previews over **plain HTTP** (zero-auth seam on `package:http`, NOT `cloud_functions` — the invitee may have no account; emulator base URL behind `USE_FUNCTIONS_EMULATOR`, routed to `demo-hayati` per the §3 emulator gotcha), mounts **pre-auth** from the sign-in screen (the invitee sees who invited them before committing to sign-in) and post-auth via `OnboardingGate` precedence **`coupleId` → pending invite → invite share screen**; the join CTA calls the §3 transactional `joinInvite` with per-`reason` typed error copy, and success re-routes through the live `users/{uid}` stream (`coupleId`-first precedence) to a paired-home placeholder (the M3 daily-question slot). The deep link is the **custom scheme `hayati://invite/<code>`** (registered in the iOS Runner, parsed by a pure `inviteCodeFromUri` into `pendingInviteProvider`; cold-start + warm links via `app_links` behind a `DeepLinkSource` seam). Universal links were not a genuine contest yet: they need Apple Developer enrollment + a hosted `apple-app-site-association` (domain choice, flagged in founder actions); upgrade path is add the associated domain + AASA later and keep the custom scheme as fallback. Remaining: M2.4 solo mode.
- **Daily rollover:** scheduled Function per timezone bucket assigns next unseen question honoring pack/register/seasonal rules; deterministic so client can prefetch offline.
- **Coach:** client → `coach_proxy` → guardrail pre-filter (crisis lexicon TR/AR/EN → localized help response path) → LLM with persona+register system prompt → post-filter → response; per-user daily budget; per-couple monthly token cap; all limits Remote Config-tunable.
- **Entitlements:** RC webhook → Function → `subscriptions/{coupleId}` → both partners unlocked by a single purchase.

## 5. Offline & performance

Firestore offline persistence ON; answers composed offline queue and reveal on sync; question of the day prefetched at prior rollover; image-light UI (mid-range Android budget: cold start <2s, first frame jank-free). Coach requires connectivity — degrade with honest empty-state.

## 6. i18n / RTL

ARB-based l10n (M1.2: sources in `app/lib/core/l10n/arb/`, EN template + TR + AR; generated output gitignored — regenerated by `flutter pub get`, verified format/analyze/coverage-safe on CI; unsupported device locales resolve to EN via `preferred-supported-locales`, mirroring the domain's content-language bootstrap); `Directionality` correctness enforced by the live six-cell golden matrix — {TR, AR, EN} × {LTR, RTL} per screen, incl. a mirror-net self-test (M1.4; `test-suite.md`); no hardcoded left/right — logical `start/end` only, lint-enforced; bundled Rubik with Noto Sans / Noto Sans Arabic fallback (OFL, `app/assets/fonts/`), body line-height 1.5 Latin / 1.7 Arabic applied per resolved locale via the design-system theme (`core/design_system/`); dates support Hijri display (v1.5 anniversaries).

## 7. Analytics schema (gate instrumentation)

`install → signup → invite_sent → paired → q_answered{solo|mutual} → reveal_viewed → streak_day → trial_start → paid → churn` plus `share_card_created`, `coach_msg`, locale/register/storefront dimensions on every event. Funnels for Gates 2–3 are built in Mixpanel before launch day, not after.

## 8. Security, privacy, compliance

App Check on all Functions; least-privilege rules; relationship content encrypted at rest (Firestore default) and **excluded from analytics payloads** (event names carry no answer text, ever); PIN/biometric lock local-only; discreet mode = alternate launcher icon (Android activity-alias / iOS alternate icons) + generic notification strings. KVKK+PDPL: consent screens per locale, self-serve export (Function → JSON email) and hard delete (cascade couple data with partner notification), DPA inventory kept in `docs/`. AI: no training on user data (provider flag), 30-day TTL on coach history, disclaimers in-product.

## 9. CI/CD

`ci.yml` on every push/PR: format → analyze → unit+widget tests → coverage gate → build debug. **`functions-rules`** (M2.1) runs **per-PR on ubuntu** (cheap minutes, unlike the macOS legs): eslint + tsc, then the Functions and `firestore.rules` suites against the auth/firestore/functions emulators via `firebase emulators:exec` (Temurin 21 on PATH for the firestore emulator; `lib/` built first because the functions emulator never compiles TS), with the vitest coverage thresholds hard-failing <80% — rules regressions are deliberately a *pre-merge* signal, and the job is a **required merge check** (branch protection: `quality` + `ios-build-smoke` + `functions-rules`) that also gates the 10×-billed macOS build via `needs`. A further job, **`integration-emulator`** (M1.3, closes ci-debt #6), boots an iOS simulator on a macOS runner and runs the `integration_test/` suites against the Auth + Firestore emulators (`firebase emulators:exec`, Java 21+, credential-free `demo-hayati`) — since M2.2 the Functions emulator too (node setup + `functions/lib` built first; `USE_FUNCTIONS_EMULATOR=true`), so `pairing_emulator_test.dart` drives the real `createInvite` callable through the app's repository. Suites run **one `flutter test` invocation each, serially** — passing the directory lets flutter_tools run them concurrently against the single simulator, where they fight over the shared `com.hayati.app` bundle. The phone suite is **quarantined with a warning annotation** (issue #15: `verifyPhoneNumber` crashes the app natively on the simulator); Google, Apple, and the profile round-trip gate the job. It is **main-push + `workflow_dispatch` only, deliberately not per-PR**: macOS minutes bill at 10× on this private repo (~100–140 billed minutes per run) and a billing failure silently blocks every merge (Session 004 incident). The accepted trade-off is that it is a *post-merge* signal — run the suites locally before merging (headers in `integration_test/*_test.dart`), or fire it on a branch with `gh workflow run ci.yml --ref <branch>`. `release.yml` tag-triggered: integration tests (iOS simulator first per iOS-first sequencing (ADR-006); Android emulator matrix added in the Android enablement follow-on, M6.5) → Fastlane build+sign → TestFlight (Play internal deferred to the Android enablement follow-on, M6.5) → store metadata per locale from `fastlane/metadata`. Pipeline-green is a merge requirement (`project-rules.md` #7). Secrets in GitHub OIDC/environment secrets; zero keys in repo.

## 10. Scalability & cost posture

Firestore fan-out is trivial at our shape (couple-scoped docs, no global feeds — a deliberate consequence of the §Social Layer decision in `prd.md`). Cost ceilings: Remote Config caps on coach usage are the only meaningful variable cost; alerting at $ thresholds via budget alerts. 100K couples fits comfortably in this architecture; the first real rework (search, community polls aggregation) is a v2 problem and is isolated behind repositories.
