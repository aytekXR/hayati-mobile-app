# Architecture — Hayati

## 1. Stack & rationale

| Layer | Choice | Why |
|---|---|---|
| App | **Flutter** (stable channel), Dart | One codebase for Android-heavy TR + iOS-heavy GCC; first-class RTL; widget-test culture fits our TDD mandate |
| State | Riverpod (+ codegen) | Testable, compile-safe DI, scales from MVP to v2 |
| Backend | **Firebase**: Auth (Google/Apple/phone), Cloud Firestore, Cloud Functions (TypeScript), FCM, Remote Config, Crashlytics, App Check | Solo-founder ops budget ≈ zero; offline persistence built-in; Remote Config gives the regional feature flags the GCC strategy requires |
| Payments | **RevenueCat** | Cross-store entitlements, webhooks, experiments; industry default for this exact app category |
| Analytics | Mixpanel (product funnels) + Firebase Analytics (audiences) | Free tiers cover Year 1; funnel tooling for the three gates |
| AI | LLM via **Cloud Function proxy only** | API keys server-side; per-user rate limits; safety filter; model-swappable; cost caps enforceable |
| CI/CD | GitHub Actions + Fastlane | Matches `project-rules.md` (gh CLI, pipeline-green rule) |

Firestore region: **europe-west (multi-region EU)** — defensible latency for both Istanbul and Riyadh, and a conservative posture for KVKK/PDPL cross-border rules. Revisit if KSA data-residency guidance tightens; the repository pattern below keeps a migration survivable.

Feature-first, clean-ish architecture: `presentation → domain ← data`. Domain layer is pure Dart (no Flutter imports) — this is what makes the 90% domain coverage target in `test-suite.md` cheap.

**Firebase environments (M1.2):** two real Firebase projects map to the two Dart-entrypoint flavors — **`hayatiapp-dev`** / **`hayatiapp-prod`** (`com.hayati.app` on both platforms; the planned `hayati-dev`/`hayati-prod` ids were unavailable). Options live as committed per-flavor Dart files (`app/lib/core/firebase/firebase_options_{dev,prod}.dart`, generated by `flutterfire configure`, FlutterFire Dart-only init — no `GoogleService-Info.plist`/`google-services.json` committed and no google-services Gradle plugin, deliberately: one Runner/one Android app serves both flavors, so platform config files can't be per-flavor; config files are not secrets, API keys get console-side restrictions). Google sign-in OAuth client ids are likewise per-flavor **runtime** config (`app/lib/core/firebase/google_sign_in_config.dart`: iOS `clientId` ≙ `GIDClientID`, Android web `serverClientId`), with both flavors' iOS `REVERSED_CLIENT_ID` URL schemes registered side by side in `Info.plist`. *Provisioning learnings (Session 004):* free-tier Firebase **Auth initialization and Google-provider enablement are console-only** — the Identity Toolkit Admin API rejects provider creation without an existing OAuth client and `identityPlatform:initializeAuth` is the billing-gated Identity Platform upgrade; everything else (API enablement, `flutterfire configure`, Firestore database creation at `eur3`, rules deploy) is scripted via `firebase-tools`/REST. Environment bindings (Firebase-backed `AuthRepository`, Firestore-backed `ProfileRepository`) are composed in the entrypoints via `runHayati(..., extraOverrides: [...])`; base providers throw when unoverridden (the `appConfigProvider` pattern). Emulators stay opt-in dart-defines (`USE_AUTH_EMULATOR`, `USE_FIRESTORE_EMULATOR`, and since M2.2 `USE_FUNCTIONS_EMULATOR` — wiring the `europe-west1` `FirebaseFunctions` instance, the same regional instance the pairing repository resolves; the Firestore emulator needs **Java 21+**, relevant to ci-debt #6); runtime verification path is `app/integration_test/`. Firebase iOS SDK 12 raised the iOS floor: deployment target pinned to **15.0** in the Runner pbxproj (iOS plugins integrate via **Swift Package Manager** — the Flutter 3.44 scaffold is SwiftPM-first and has no Podfile; adding one breaks the build, verified on CI in Session 003).

**Auth providers (M1.3).** Three providers reach Firebase through one uniform seam in `features/auth/data/`: a gateway per provider that yields a Firebase `AuthCredential` (or `null` when the user cancels), which `FirebaseAuthRepository` exchanges via `signInWithCredential` inside its `_guarded` boundary. This shape exists because `signInWithCredential` forwards the credential unvalidated, so the Auth emulator accepts an *unsigned* JSON `id_token` — which is what makes every provider testable without a real account ([ADR-008](adr/008-apple-signin-via-credential-seam.md)). **Apple** uses `sign_in_with_apple` + `AppleAuthProvider.credentialWithIDToken` (never `signInWithProvider`, which surfaces no credential and therefore cannot be faked or emulated); the nonce is a two-value protocol — SHA-256 hex to Apple, plain `rawNonce` to Firebase. **Phone** is the one flow with a user-input gap, so it cannot be a single atomic call: `PhoneAuthGateway` adapts `verifyPhoneNumber`'s four callbacks into one `Completer` (guarded against double-completion — `verificationFailed`/`codeAutoRetrievalTimeout` can fire *after* `codeSent`, and the returned Future can reject independently), and a screen-scoped `PhoneSignInController` drives enter-number → code-sent → confirm. That controller **never writes the global `AuthState`**: the terminal `AuthSignedIn` arrives on the `authStateChanges` stream while `AuthController` is idle, preserving its manual-op precedence contract. On iOS `verificationCompleted` never fires (Android-only), so a code-entry UI is mandatory; on Android it fires *instead of* `codeSent` under instant verification, which would hang the send forever — the gateway therefore fails loudly there until real Android support lands (issue #13, M6.5). Emulator-side, phone codes are minted randomly per request and read back over the emulator's REST endpoint — **under the project the emulator was booted with (`demo-hayati`), not the app's `FirebaseOptions.projectId`** (the Auth emulator resolves every SDK request to its single `--project`; querying `hayatiapp-dev` returns an empty list, verified 2026-07-09).

**Crashlytics & App Check (M1.3).** Both are activated **only from the flavor entrypoints**, after `initializeFirebase` — never inside it, and never from `HayatiApp` — because `FirebaseAppCheck.activate()` and `FirebaseCrashlytics.instance` each drive a platform channel that throws in the plain `flutter test` VM, where the bootstrap and widget trees are unit-tested. Crash reporting crosses into the app behind a `CrashReporter` interface taking only framework types (`core/observability/`); exactly one file imports `firebase_crashlytics`, and `runHayati(..., crashReporter:)` installs `FlutterError.onError` + `PlatformDispatcher.onError` hooks only when a reporter is supplied, so tests stay channel-free. **Per-flavor collection policy: dev OFF (noise), prod ON**, applied at boot via the runtime `setCrashlyticsCollectionEnabled` API (it persists across launches and overrides plist defaults, so a device that switched flavors self-corrects). **App Check posture:** dev activates the *debug* provider, prod declares *App Attest* (always available at the min iOS 15.0 floor); activation is skipped entirely under `kUseAuthEmulator` so the integration leg stays hermetic. **Enforcement must stay OFF in both consoles until on-device attestation is verified** — enabling it early would lock the founder couple out of Auth/Firestore. Sign in with Apple adds `app/ios/Runner/Runner.entitlements` (`com.apple.developer.applesignin`), wired via `CODE_SIGN_ENTITLEMENTS` in the three Runner build configs; entitlements are consumed at codesign time, so CI's `--no-codesign` build is unaffected. *Deferred loudly to the on-device/Mac slice (nothing silent, rule #9):* App Attest entitlement + console app registration, per-device App Check debug-token registration, the `FirebaseCrashlyticsCollectionEnabled=false` Info.plist baseline that would close dev's first-launch pre-Dart collection window, dSYM upload for prod symbolication (the CocoaPods podspec adds it; SwiftPM does not), and real-device verification that Apple's first-authorization full name reaches `displayName`.

**Security rules invariants (tested against emulator):** a user reads/writes only their own couple's docs; partner's answer for today is unreadable until requester's answer exists (mutual-reveal enforced server-side, not just UI); invites expire; entitlement docs are function-write-only. *Shipped so far (M2.1 → M3.3, `firestore.rules`):* `users/{uid}` self-only with granular ops — `createdAt` **rules-enforced** create-once (`== request.time` at create, frozen on update — the M1.2 client transaction shape passes unchanged), client delete denied (hard delete is the M6 cascade Function), and since M2.3 **`coupleId` frozen against clients** (create may not include it, update may not set/change/delete it — `get('coupleId', null)` compared symmetrically so the app's merge writes pass; the field is authorization-meaningful, granting `couples/` access via `memberUids`, and is written only by the join Function); since M2.4 **`users/{uid}/soloAnswers/{dayKey}` self-only with a frozen field surface** — `hasOnly([questionId, text, answeredAt])`, bounded non-empty text (≤2000), `answeredAt == request.time` (a client clock can never forge the answer time; same server-stamp discipline as `createdAt`), same-day overwrites allowed by design (the answer stays editable all day), client delete denied (the M6 cascade owns removal); `couples/{coupleId}` member-only read/update with `memberUids` frozen (couple create/delete is exclusively the M2.3 transactional-join Function / M6 cascade) — **and since M3.3 `timezone` + `createdAt` frozen too** (the stored zone is dayKey-load-bearing on BOTH sides: the rollover buckets by it and the app keys the day by it, so a member rewriting it to junk would brick the couple's loop; zone changes move behind a future settings Function, mutation-tested both freezes); since M3.2 **`couples/{cid}/days/{dayKey}` member-only read with ALL client writes denied** (function-only via admin SDK — the rollover is the sole writer; membership needs a `get()` of the parent couple because the day doc carries no uids, and an orphaned day under a deleted couple fails closed), mutation-tested both ways (membership guard weakened → non-member read succeeds; the write-deny swapped for authed-allow → client write succeeds); since M3.3 **`couples/{cid}/days/{dayKey}/answers/{authorUid}` carries the mutual-reveal invariant server-side**: create/update self-only AND member-of-couple (parent `get()`), frozen `hasOnly([questionId, text, answeredAt])` surface with non-empty text ≤2000 and `answeredAt == request.time`, `questionId` pinned by `get()` to the parent day doc's assigned question (no day doc → no answers, fails closed), **read = member AND (own doc OR `exists()` of the requester's own answer)** — the partner's answer is unreadable before the requester answers, a whole-collection `list` flips with the same branch (denied pre-answer, allowed post-reveal), updates are denied once BOTH answers exist (reveal freezes the pair — no rewrite-after-reading), delete denied (the reveal gate is one-way; removal is the M6 cascade). Proven by 21 new rules cases plus **12 mutation tests** (reveal `exists()` gate dropped → pre-answer partner read succeeds; read/write membership guards dropped → non-member access succeeds; `isSelf` weakened → partner-doc writes succeed; `hasOnly`/server-stamp/questionId-pin/non-empty-text dropped → each junk write succeeds; both-answered freeze dropped → post-reveal rewrite succeeds; delete allowed → post-reveal retraction succeeds; `timezone`/`createdAt` freezes dropped → member rewrites succeed); since M3.4 **`couples.streak` frozen too** (same symmetric `get('streak', null)` comparison as `users.coupleId` because the field is absent pre-first-mutual-day — a member can neither introduce, change, nor delete it; the answerReveal trigger is its sole writer, mutation-tested like the other freezes); `invites/{code}` **no client access at all** — writes come from `createInvite`/`joinInvite` (admin SDK) and the invitee-facing read is the M2.2 zero-auth preview Function, so codes can't be enumerated client-side; since M4.1 **`subscriptions/{coupleId}` member-only read with ALL client writes denied** (the doc id IS the coupleId, so membership needs a `get()` of that couple — the M3.2 `days` pattern; an orphaned mirror under a deleted couple fails closed; the `revenueCatWebhook` Function is the sole writer, delete belongs to the M6 cascade), mutation-tested both ways (write-deny swapped for authed-allow → member client write succeeds; membership guard weakened → non-member read succeeds); default-deny for everything else. Every invariant is proven by `functions/test/emulator/firestore-rules.test.ts`, including **mutation tests** that load a weakened copy of the rules and show the denial disappears — the suite goes red if a protecting clause is commented out. Client wire mapping for `users/{uid}` lives in `app/lib/features/profile/data/profile_dto.dart` — enum `name`s are the wire values and renaming one is a data migration.

**Invite issuing (M2.1, `functions/`).** The repo's first Cloud Functions code: TypeScript strict on **firebase-functions v7 / Node 20** (firebase-admin pinned `^13` — v14 requires Node ≥22), vitest + v8 coverage with the `test-suite.md` §3 gate (target 85%, **hard fail <80%** via thresholds), eslint flat config. `createInvite` is an **onCall v2 callable in `europe-west1`** (closest single region to both Istanbul and Riyadh, consistent with eur3; the M2.2 client must construct its callable with this region): auth required (guarding `auth.uid` itself, not just presence — the functions emulator skips real token verification, and it's free defense-in-depth), App Check **token plumbed but enforcement OFF** (`enforceAppCheck: false` until on-device attestation is verified, M1.3 posture; token presence is logged to observe attestation coverage pre-enforcement). Codes are 8 chars from a 31-char unambiguous alphabet (A–Z + 2–9 minus O/I/L — survives being read aloud or retyped from WhatsApp; 31⁸ ≈ 8.5e11). **Re-issue policy: one active invite per creator** — a pending, unexpired invite is returned as-is (idempotent; re-opening the share flow never invalidates the code the partner already has), stale pendings are lazily marked `expired` and replaced; invites never accumulate. Everything runs in one Firestore transaction (pending-invite query + collision probes are reads-before-writes; `tx.create` fails if the code doc appeared meanwhile), so concurrent calls serialize and the loser returns the winner's code — proven against the emulator. Emulator wiring gotchas that cost real time, recorded once: the functions emulator **never compiles TS** (build `lib/` before `emulators:exec`), injects **no functions-host env var** into the exec child (tests derive `http://127.0.0.1:5001/<project>/<region>/<name>` from the pinned port), firebase-tools finds Java **via PATH only** (JAVA_HOME is never read; the firestore emulator needs Java 21+), and the functions emulator serves functions **only under its `--project` id** — a Flutter app initialized with a real flavor's `FirebaseOptions` builds the emulated callable URL from *its* `projectId` and 404s (`NOT FOUND`), because unlike the auth/firestore emulators (which resolve any project id) the functions emulator routes by URL path; emulator-backed integration tests must therefore initialize their default app with `demo-hayati` options (found the hard way on the first `integration-emulator` run of M2.2, fixed in PR #23).

**Invite preview (M2.2, `functions/`).** `invitePreview` is an **onRequest v2 HTTP endpoint in `europe-west1`**, zero-auth by design — the invitee has no account yet, so a callable's auth/App Check envelope would be dead weight (the onCall-vs-onRequest split was pre-decided in Session 007, not re-contested). Contract: `GET ?code=` → HTTP 200 `{status: 'valid'|'expired'|'unknown', creatorDisplayName?}`. **The response surface is exactly those fields** — never `creatorUid`, other invites, or (once M3 content exists) answer content; the suite's field-surface tests assert `Object.keys(body)` per case, so an accidentally added field turns CI red. This is where the §4 invariant "preview never leaks answer content" starts; the projection lives in one typed helper so the M3 growth (`questionText` + locked indicator) stays an explicit, reviewed surface change. Lookups return a **uniform 200/`{status}` shape for all three outcomes** (no enumeration oracle beyond validity — `expired` vs `unknown` only distinguishes a code you already hold), and malformed codes (checked against the alphabet/length imported from `invite-code.ts`) short-circuit to `unknown` without a Firestore read. The endpoint is strictly **read-only**: a stale pending is *reported* `expired` by comparing `expiresAt`, never written — a zero-auth path must not amplify writes; lazy expiry stays `createInvite`'s job (proven by a test that previews a stale invite and re-reads the doc). `creatorDisplayName` comes from the creator's **Auth record** (`users/{uid}` stores no display name) and is omitted — still `valid` — on any lookup failure. Logs avoid becoming an oracle: a 3-char code prefix is logged, never validity-plus-full-code. **Rate-limiting decision (recorded per resume-prompt):** a best-effort **in-memory fixed-window per-IP limiter (30 req/min → 429)** guards the endpoint — per-instance and reset on cold start, so it is an abuse damper, not a guarantee; the real anti-enumeration defense is the 31⁸ (~8.5e11) code space plus function-only reads. Infrastructure-grade limiting (Cloud Armor / API Gateway) is deliberately out of scope until a public launch surface exists. No CORS: the native app is the only consumer; a future web preview adds it deliberately.

**Transactional join (M2.3, `functions/`).** `joinInvite` is an **onCall v2 callable in `europe-west1`** (auth required with the same uid-not-just-presence guard as `createInvite`; App Check plumbed, enforcement OFF — M1.3 posture). Request `{code, timezone?}` → `{coupleId}`. Everything happens in **one Firestore transaction** (all reads before writes): read `invites/{code}` → both `users` docs, check in a deliberate order (existence → terminal-state → self-join → profile-missing → already-paired), then create `couples/{auto-id}` (`memberUids: [creatorUid, joinerUid]` — creator first, frozen), set `coupleId` on both `users` docs, and stamp the invite `{status: 'joined', coupleId, joinerUid, joinedAt}`. **Typed error surface (frozen wire contract the app maps on):** `not-found`/`{reason:'unknown'}` for absent or malformed codes (malformed short-circuits without a read — no enumeration oracle); `failed-precondition` with `reason` `'expired'` (status or clock or malformed doc), `'consumed'` (status `'joined'` — the race loser's answer), `'self-join'`, `'already-paired'` (either side has `users/{uid}.coupleId`), `'profile-missing'` (either `users` doc absent — the app flow guarantees profile-before-join; this is defense-in-depth). **Race semantics are the acceptance criterion:** two concurrent joins serialize on the invite doc; the admin SDK retries the ABORTED loser, whose retry re-reads the invite — now `'joined'` — and throws `consumed`; typed rejections always come from re-read state, never from inspecting commit conflicts (proven at service AND callable level against the emulator, exactly one couple per code). **Couple `timezone` source (documented decision):** the joiner MAY supply an IANA id, validated against `Intl.supportedValuesOf('timeZone')`; absent/unrecognized falls back to `Europe/Istanbul` (founder-couple / TR soft-launch home zone). The app currently sends nothing — a reliable device IANA source needs a plugin, deferred to M3 where the rollover work makes it load-bearing. `createInvite` gained the symmetric guard: an already-paired creator gets `failed-precondition`/`'already-paired'` instead of a fresh code, so re-issue after a join can never resurrect or replace a spent code (a `'joined'` invite also previews as the uniform `'expired'`).

**Solo mode (M2.4, app-side — no Functions footprint; full rationale in ADR-009).** The unpaired user's home shows the day-N solo reflection question with an answer entry and a **persistent invite nudge** on every settled state (honest better-together framing per `prd.md` F1). Content: three bundled JSON packs (`app/assets/content/solo_{tr,ar,en}.json`) shaped exactly like `content/schema/question-pack.schema.json` (register `neutral`, exactly 7 questions — enforced at load), selected by `profile.contentLanguage` and read through a `SoloQuestionPackRepository` seam with a loud pure mapper. **Since M3.1 the packs are authored under `content/packs/` and byte-synced into the bundle by the enforcing validator (ADR-010: CI check mode red on drift, `--sync` regenerates), and the model is the general one** — `QuestionPack`/`Question` (`register` carried, per-question `seasonalWindow` surfaced) loaded by packId through `AssetQuestionPackRepository`, the solo path a thin specialization (`solo_<locale>`, exactly-7 + locale checks) behind the unchanged seam; remote pack sync and register-aware resolution remain M3.2+. Day N anchors on `users/{uid}.createdAt` — surfaced to the client **READ-ONLY** exactly as `coupleId` was at M2.3 (the repository converts the wire `Timestamp` to `DateTime` at the boundary; `profileToMap` still never emits it) — and is a **local calendar-date distance** (`solo_day.dart`: date components only, so DST can't move a boundary; null/future anchors clamp to day 1; the wall clock enters via `soloClockProvider`, the app's single clock seam). **Day 8+ stops the cycle** — questions never repeat; the completed state makes the invite the primary action. Answers persist to `users/{uid}/soloAnswers/{yyyymmdd}` (dayKey = the same local calendar date; same-day saves overwrite — the answer stays editable all day) through a `SoloAnswersRepository` seam mirroring the profile repository's exception taxonomy. The share flow stays one tap away behind the nudge as a **pushed route** that pops itself when `coupleId` lands (partner joined while the user sat on it) or on sign-out — the gate underneath has already re-routed either way.

**Daily rollover (M3.2, `functions/` — full rationale in ADR-011).** `questionRollover` is the repo's first **scheduled Function** (`onSchedule` v2, `europe-west1`, `0 * * * *` UTC, explicit `timeoutSeconds`/`memory`): a **single hourly sweep** reads all couples, buckets them by their stored `timezone` (verbatim — never re-resolved through the join-time default, so corrupt state surfaces instead of masquerading as Istanbul), computes each bucket's **local calendar date once** (`rollover/day-key.ts`: `Intl.DateTimeFormat` `formatToParts` pinned to `gregory`/`latn` — DST-correct by construction, immune to locale calendar/digit drift), and **creates `days/{dayKey}` only if absent** (`create()`; an existing day doc is never reassigned, losing the race to an overlapping run is a benign no-op because selection is deterministic). Selection (`rollover/select-question.ts`, pure): first **unseen** evergreen question in pack authoring order, then min-times-assigned recycle with pack-order tie-break after exhaustion; **seasonal questions are excluded entirely** — the documented evergreen-only deferral until the Hijri window→date mapping lands with the first seasonal content. History = a `questionId` projection over the couple's `days` subcollection (O(history) reads/assignment; the scale path — timezone-indexed sweep query + selection cursor — is documented in ADR-011 and deliberately not built). Packs are **bundled at build time** (`npm run build` = `tsc` + `scripts/bundle-packs.mjs` copying `content/packs/` → `lib/content/packs/`, byte-equality suite-enforced; the validator stays the single content gate, the strict TS parser at the consumption edge is defense-in-depth). **Error boundary:** per-couple problems (malformed `packConfig`, missing/corrupt timezone, unloadable pack, all-seasonal pack) are logged skips counted in the run summary — never a failed run; the throw is reserved for systemic failures. The handler validates `event.scheduleTime` (nominal scheduled instant beats wall clock; garbage falls back loudly). The schedule *trigger* is emulator-untestable and **deploy-verified later** (Blaze item, `operator-expected.md`); everything else is proven in-process against the emulator — timezone buckets incl. DST boundaries, double-run idempotency at service AND handler level, the create race, and the per-couple skip taxonomy.

**Mutual reveal (M3.3, app + rules — the couple's daily loop closes).** The paired home (`features/daily_question/presentation/paired_home_screen.dart`, replacing the M2.3 placeholder behind the gate's `coupleId` route) walks the read path `couples/{coupleId}` (the app's first couple read — watch-only `CoupleRepository`; the doc carries the zone) → **`coupleDayKey(now, couples.timezone)`** (`couple_day.dart`, the Dart mirror of the rollover's `localDayKey` over `package:timezone`'s 10-year db, initialized at bootstrap and lazily as a net; parity with TS pinned byte-for-byte by the shared fixture `functions/test/fixtures/day-key-parity.json` incl. DST boundaries and sub-hour zones — NEVER `soloDayKey`/the device zone, ADR-011) → `days/{dayKey}` (watch-only; **null renders the honest no-day-yet state** and streams the assignment in live when the hourly sweep lands it — the client never predicts) → question text resolved from the bundled pack by `packId`+`questionId` (`AssetQuestionPackRepository`; an id the install doesn't bundle renders the "update the app" pack-lag state, typed via `UnknownQuestionPackException`). Answers write through `CoupleAnswersRepository.saveAnswer` (full-replace `set` of exactly the rules surface; `answeredAt` = `FieldValue.serverTimestamp()`). **The client half of the reveal invariant is `partnerSlotProvider`** (sealed `PartnerSlot`: locked → waiting → revealed | failure): it never attaches the partner-answer watch until the OWN answer is **server-acked** (`answeredAt != null` — a pending serverTimestamp local echo crosses as null), because a pre-answer listen would be denied and the SDK never retries a denied listen; a denial that slips through anyway (lost `exists()` race on a replica) maps to the locked state as defense-in-depth plus a bounded 3× re-subscribe (1s/2s/4s — the ONE narrow retry in the couple providers; everything else keeps the no-retry philosophy: corrupt state hammers nobody). Both-answered is the revealed state: the entry collapses to a read-only own-answer card (rules have frozen both docs). The screen recomputes the dayKey per build from `soloClockProvider` and on app-resume (`WidgetsBindingObserver` — the dominant path back after a couple's midnight; still no timer, ADR-009 precedent). A corrupt stored zone (`LocationNotFoundException`) renders an honest error view, never a guessed date. Couple questions come from the `solo_tr` placeholder bank until W9 (ADR-011) — the founders will recognize their solo questions; accepted for dogfooding.

A transfer is a bare pointer (no product, no expiry, no entitlement ids) so **it never creates or entitles a GAINER's lane** (the only honest expiry would be the null=non-expiring sentinel — permanent free premium); the gainer's couple stays free until its next lifecycle event. The LOSS half revokes **only on positive evidence the entitlement left the couple**: ids resolve per-id (no alias chain), an anon id in `transferred_from` is filtered but an anon id **anywhere in `transferred_to` HOLDS** (it may be the same human mid-reinstall — revoking there would strip premium from a paying couple), a within-couple transfer is a no-op (the mirror is couple-scoped: only which RC subscriber holds the entitlement changed), and every other ambiguity holds. A pre-read gate (filter → dedupe → cap 10/array) bounds a hostile payload to ZERO reads. Revocation writes a **tombstone** that is a pure projection of the transfer alone (`expiresAtMs` = the transfer instant, never null) — never a copy of the loser's prior lane, which would break convergence — guarded by the same `(ts, id)` LWW order key, one transaction per target couple (partial application is safe by idempotence, not atomicity). Deliberately absent, recorded in ADR-013/014/015: rate limiting (a 429 burns RC's finite 5-retry budget on real events), RC-API reconciliation/backfill (a webhook dropped past RC's ~155-minute retry budget has no recovery until the deploy-era backfill — it is also the fix for the transfer design's three named costs), identity-sync cursor-reset-on-failure hardening (first live-key session). **Gifting (PRD F4) is not a payments feature** (ADR-015 Decision 6): no StoreKit gifting API and no App Store Connect giftable toggle exist for auto-renewable subscriptions, and Guideline 3.1.1 permits buyer≠beneficiary by name — so the couple-scoped mirror IS the gift, no gift UI ships, and Family Sharing stays OFF on the products (irreversible once enabled).

the OS app-switcher snapshot (the system imaging the on-screen conversation to disk on backgrounding) is a documented residual exposure whose mitigation (snapshot obscuring) rides M6's device-privacy layer with PIN/biometric + discreet icon. ## 10. Scalability & cost posture

Firestore fan-out is trivial at our shape (couple-scoped docs, no global feeds — a deliberate consequence of the §Social Layer decision in `prd.md`). Cost ceilings: caps on coach usage are the only meaningful variable cost (M5.1: code-constant defaults behind an injectable seam, Remote-Config-tunable at the deploy era); alerting at $ thresholds via budget alerts. The M5.1 `coachProxy` cost shape (ADR-016): a crisis/help-path call costs ZERO reads and zero provider spend; a persona call is one bounded transaction (4 doc reads, 2 writes) + one provider call, pre-paid by a reserved cap unit — the per-uid 30/min limiter, the 30/day/user cap, and the 1,000/month/couple cap bound worst-case provider spend to ≈$14/couple/month at Sonnet-tier pricing before any key even exists. The M3.2 rollover's cost shape (ADR-011): O(couples) doc reads per hourly sweep plus an O(days-history) projection read per assignment — pennies for years at personal-use scale; the binding fleet-scale constraint is the invocation wall-clock (sequential per-couple reads), with the timezone-indexed sweep query + selection cursor as the documented rework path. The M3.4 additions keep that shape (ADR-012): `answerReveal` fires once per answer create (≤2/couple/day) with an O(1) transaction (4 doc reads + ≤2 writes) plus ≤1 users-doc read per push; the at-risk pass reuses the sweep's single couples read (shared timezone buckets — the sweep is still ONE couples query per hour) and adds one day-doc read per streak-holding couple only in its hour-20-local bucket, once per day. The M4.1 webhook is O(1) per event (ADR-013): ≤3 user-doc reads for identity resolution plus one bounded transaction (1 mirror read, 1 write) — RC event volume is ≤ a handful per couple per month. A `TRANSFER` (M4.3, ADR-015) is bounded by the pre-read gate: ≤20 identity reads (10 per array, after a pure anon-filter/dedupe) and ≤1 transaction per resolving loser (1 in practice); an oversized or unplaceable transfer costs ZERO reads. 100K couples fits comfortably in this architecture; the first real rework (search, community polls aggregation) is a v2 problem and is isolated behind repositories.

[ADR-001](adr/001-flutter-over-native.md): Flutter over native pair. [ADR-002](adr/002-firebase-over-supabase.md): Firebase over Supabase (offline + FCM + Remote Config maturity outweigh Postgres ergonomics for this shape). [ADR-003](adr/003-revenuecat-for-entitlements.md): RevenueCat. [ADR-004](adr/004-eu-firestore-region.md): EU region. [ADR-007](adr/007-de-gate-build-from-content-validation.md): build de-gated from content validation — M1→M6 proceed without content-ops preconditions; Gates 1–3 become marketing/spend/launch decision instruments; personal-use-first (founder couple) is the first release target. [ADR-008](adr/008-apple-signin-via-credential-seam.md): Sign in with Apple through a credential seam (`sign_in_with_apple` + `signInWithCredential`), rejecting `signInWithProvider` because it surfaces no credential and is therefore neither fakeable nor emulator-testable. [ADR-009](adr/009-solo-mode-content-and-persistence.md): solo mode — bundled schema-shaped packs, `createdAt`-anchored local calendar-day rotation, Firestore `soloAnswers`. [ADR-010](adr/010-content-authoring-home-and-sync.md): `content/packs/` as the single question-pack authoring home with validator-owned one-way byte-sync into `app/assets/content/` (check mode never writes; CI red on drift; example pack deleted).
