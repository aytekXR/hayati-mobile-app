Milestones are sized in **sessions** (one session = one `resume-prompt.md` objective, per `project-rules.md`). TDD per `test-suite.md` throughout: acceptance criteria below are expressed as tests wherever possible.

**Release sequencing (iOS-first, ADR-006):** M0–M6 build the single Flutter codebase (`architecture.md` ADR-001 stands) but validate and ship on **iOS first** — simulator/integration tests, iOS goldens, App Store sandbox, TestFlight, closed beta + TR soft launch on iOS. One codebase keeps the Android follow-on cheap while concentrating initial release risk on a single platform.

M4.3 ✅ *(Session 017, 2026-07-12)* — `TRANSFER` handling + the gift decision (ADR-015), emulator-only by design (operator item 0 still absent — the session took the planned engineering fork). **Headline finding, caught by re-reading RC's live docs: a real `TRANSFER` was a 400, not the "logged no-op 200" ADR-013 assumed.** RC's transfer body carries **no `app_user_id`** (the subscriber-identity field group does not apply to it — `transferred_from[]`/`transferred_to[]` instead), so `parseRcEvent` classified every genuine transfer `malformed` → 400 → RC burned its 5-retry budget over ~155 min and **dropped the event permanently**. No test caught it because every TRANSFER test built a *post-parse* `RcEvent` and both raw-envelope builders injected `app_user_id` unconditionally — the envelope contract had never seen a real transfer body. The envelope contract is now **per-type** (`app_user_id` still required for every LIFECYCLE type — a lifecycle event without one is still a 400; the exception is exactly one type wide), and an e2e wire test POSTs RC's documented sample body. **Two facts drove the whole design:** a transfer is a *bare pointer* (no product, no expiry, no entitlement ids, and RC sends exactly ONE webhook with **no follow-up `EXPIRATION` for the loser**), so (1) **the gain half is structurally unprojectable** — the only available expiry is the `null` non-expiring sentinel, i.e. permanent free premium (the null-grace `BILLING_ISSUE` bug class ADR-013 already killed once) ⇒ **a TRANSFER never creates or entitles a gainer's lane**, ever; and (2) the loss half revokes **only on positive evidence the entitlement left the couple** — because a false revoke instantly strips a paying couple (unrecoverable without a manual grant we cannot issue) while a missed revoke only leaks the tail of a period someone actually paid for, and self-retires against the app's `entitled && unexpired` check. **Every ambiguity therefore holds:** within-couple transfers are a no-op (the mirror is *couple*-scoped — a shared-Apple-ID restore changes only which RC subscriber holds the entitlement, and the doc is proven BYTE-unchanged); an anon id in `transferred_from` is filtered, but an anon id **anywhere in `transferred_to` HOLDS** (the reinstall trap: delete app → RC mints an anonymous id → the store auto-restores → *then* `logIn(uid)` — the destination may be the same human, and revoking there downgrades a payer on a reinstall); an unplaceable destination holds; a loser that resolves to nothing holds. Ids resolve **per-id** (no alias chain — RC never commits to whether an array is one customer's aliases or several customers, and that premise cuts both ways), behind a **pre-read gate** (filter → dedupe → cap 10/array) that costs a hostile payload **ZERO** Firestore reads. Revocation writes a **tombstone that is a pure projection of the transfer alone** (`expiresAtMs` = the transfer instant — honest, and never the non-expiring sentinel), never a copy of the loser's prior lane: a lane value derived from another lane value stops being the projection of the total-order-maximal event and the mirror stops converging — which is also why the intuitive "MOVE the lane to the gainer" design is rejected (reordered, it drops a paying couple to free). It is written **even when no lane exists**, so a transfer arriving *before* the purchase it moves cannot be resurrected by the late (older-ts) purchase. Same `(event_timestamp_ms, event.id)` LWW guard as every other lane; **one transaction per target couple** (the couple doc is the aggregate boundary — partial application is safe by IDEMPOTENCE, not atomicity). **The convergence property now folds a TWO-COUPLE world with transfers in the multiset and is mutation-verified**: copying the loser's facts, entitling a gainer, and dropping the tombstone-when-absent rule each turn it red. **The gift decision (PRD F4 rewritten):** gifting an auto-renewable subscription **is not an App Store feature** — no StoreKit gifting API, no ASC giftable toggle; Apple's own path is a gift-card top-up so the recipient buys it themselves. Guideline 3.1.1 permits buyer≠beneficiary **by name** provided the money moves through IAP. **So the couple-scoped mirror IS the gift, F4 is already shipped, and NO gift UI ships** — a "Gift Premium" button would imply a SKU that cannot exist, and any gift entry point is precisely what would need a paywall without a `coupleId`, reopening the purchase-before-pairing gap ADR-014 structurally closed. What ships instead: the decision, the PRD rewrite, and a **regression test pinning the promise** (the non-purchasing partner unlocks off their partner's purchase with an empty purchases call log and `PaywallScreen` never mounted). Family Sharing is recorded as **DO NOT ENABLE** — irreversible once on, and it would create a second entitlement source the mirror does not own (operator item 0). Rejected and recorded so they are never re-litigated: offer codes (developer-minted only; on an auto-renewable sub they **auto-bill the recipient** at full price after the offer — a "gift" that silently charges your partner), RC promotional entitlements (a support tool, no receipt/renewal), a web/Stripe gift link (3.1.1). Apple **Group Purchases** (WWDC26, no RC support yet) is the only thing that would reopen this — tracked, not built. Functions suite **534 tests / 98.05% stmts / 95.86% branches** (gate 80); app **901 tests / 86.50%**, **coverage ratchet 66→68 landed**, analyze clean. Design adversarially reviewed BEFORE implementation (4 lenses → 2 skeptics per finding): 13 raised, 10 refuted, **3 confirmed and folded into ADR-015 pre-code** — the 10-id cap had no defined enforcement point (it now gates *before* the reads it was supposed to bound, and filters/dedupes first so routine anon-alias accretion cannot make the revoke path inert), and the `to`-side anon filter **contradicted the ADR's own "FULLY KNOWN" rule** on the false-downgrade side (fixed: any anon in `transferred_to` holds).
