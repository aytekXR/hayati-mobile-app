# Test Suite Strategy ‚Äî Hayati

Strict TDD (rule #6). The pyramid, tooling, and gates below are enforced by CI, not by good intentions.

## 1. Layers

### Unit tests (`test/`, mirrors `lib/`) ‚Äî the wide base
Pure-Dart domain layer makes these cheap and fast. Mandatory targets: streak engine (property-based: gaps, grace tokens, timezone edges, DST, Ramadan quiet-window interactions); pairing state machine (races, expiry, re-pair attempts); question selection (register/seasonal/no-repeat determinism); entitlement mapping (one purchase ‚Üí two partners; expiry ‚Üí both); pack validator; crisis-lexicon matcher (TR/AR/EN seed lists). Tools: `flutter_test`, `mocktail`, fakes over mocks for repositories, builder-pattern test data (`aCouple().paired().withStreak(12).build()`).

### Widget tests ‚Äî every screen, every language, both directions
Each screen: states (loading/empty/error/content) + interaction contracts. **Golden tests per key screen in a 6-cell matrix: {TR, AR, EN} √ó {LTR, RTL-where-applicable}, plus dynamic-type 130% variants for onboarding, question card, paywall, reveal.** **Live since M1.4** for the M1 auth/profile screens (sign-in, phone sign-in, onboarding gate, profile capture, invite placeholder), including the 130% dynamic-type onboarding variant and a mirror-net self-test that fails the build if an arrow renders un-mirrored in RTL. M2.2 replaced the invite placeholder with the real **invite share screen** ‚Äî goldens cover has-code / loading / error across all six cells (deterministic fixed code + expiry). Goldens are the RTL regression net ‚Äî an un-mirrored arrow fails the build, not the Riyadh user. Golden updates require an explicit flag in the PR (see `agent-workflows.md` W4).

### Integration tests (`integration_test/` + Firebase Emulator Suite)
Security rules suite (the invariants in `architecture.md` ¬ß3 ‚Äî especially *partner answer unreadable before own answer* and *preview endpoint never leaks answer content*); auth‚Üípair two-device flow; answer‚Üíreveal‚Üíthread round trip; delete-cascade with partner notification; RevenueCat sandbox/StoreKit-test purchase ‚Üí entitlement mirror; coach_proxy contract tests against recorded LLM fixtures + live safety tests (seeded crisis phrases in all three languages must route to the help path).

### Functions + rules tests (`functions/test/`, M2.1) ‚Äî vitest against the Emulator Suite
**Tooling decision (Session 007): vitest** over jest ‚Äî TS-native via esbuild, first-class v8 coverage with `thresholds` as the ¬ß3 hard gate (proven to exit non-zero below 80%), one config. `fast-check` for property tests (code charset/length/collision). Because esbuild strips types **without checking them** and the build tsconfig covers only `src/`, `npm run typecheck` (`tsconfig.test.json`, bundler module semantics) is the step that typechecks `test/` ‚Äî wired into the CI job. Three layers, all in one `firebase emulators:exec --only auth,firestore,functions --project demo-hayati 'cd functions && npm run test:ci'` run: **pure unit** (no emulator), **service/handler in-process** against the firestore emulator (admin SDK, injectable seams for error paths), and **callable end-to-end** through the functions emulator (real HTTP callable protocol + auth-emulator idTokens ‚Äî note the functions emulator *skips* real token verification, so production token rejection is untestable here; handlers guard `auth.uid` themselves). The **rules suite** (`@firebase/rules-unit-testing` v5) proves every shipped invariant and adds **mutation tests**: each protecting clause is weakened in a copy of the rules (loaded under its own `demo-*` projectId ‚Äî the emulator stores rules per project) and the previously-denied op must then succeed, so a commented-out rule turns the suite red instead of silently shrinking the net. Runs per-PR in the ubuntu `functions-rules` CI job (`architecture.md` ¬ß9). **M2.2 added the `invitePreview` suite** in the same three layers: pure unit (code normalization), in-process handler tests (rate-limiter semantics under an injected clock; error mapping; auth-lookup-failure degradation ‚Äî also the layer that covers the real Auth-lookup path, since v8 coverage can't observe the separate functions-emulator process), and onRequest end-to-end ‚Äî where **every 200 response carries a field-surface assertion** (`Object.keys` equals the documented set exactly, body never contains `creatorUid`) plus a read-only proof (previewing a stale invite leaves the doc `pending`). The rate-limit e2e fires its burst *sequentially* and *last in the file* (61 concurrent GETs overwhelm the emulator's per-request worker loader, and the per-IP window it fills would starve any test after it). **M2.3 added the `joinInvite` suite** in the same three layers (unit: timezone allow-list + code-normalizer relocation into `invite-code.ts`; in-process service/handler: every typed rejection, the deliberate check order, and the **concurrent-race proof** ‚Äî `Promise.allSettled` double-join ‚Üí exactly one couple, loser gets `consumed` from re-read transaction state; callable e2e: the same race through the real callable protocol plus the per-error `details.reason` surface) and extended the rules suite with mutation tests for both new `users.coupleId` freeze clauses (create-may-not-include, update-may-not-change) ‚Äî the weakened-rules-copy harness proves each denial disappears when its clause is removed. Re-issue-after-join is pinned by tests on both callables: the spent code stays `'joined'` and the paired creator gets `already-paired`.

### End-to-end (Patrol) ‚Äî critical paths only, release-gated
E2E-1 fresh install ‚Üí signup ‚Üí invite ‚Üí (second device) preview ‚Üí join ‚Üí both answer ‚Üí reveal ‚Üí streak=1. E2E-2 trial start ‚Üí sandbox purchase ‚Üí premium unlocked on *both* devices. E2E-3 PIN lock + discreet icon behavior. Run in `release.yml`, not on every push (cost/flake budget).

### Smoke (post-build, physical devices)
Signed artifact checklist on iPhone in TR + AR locales (TR = the iOS soft-launch cohort, AR = RTL sanity): cold start <2s, push received, purchase sheet opens, RTL sanity, crash-free session. 10 minutes, every release candidate. **iOS-first (ADR-006): the iPhone smoke gates the release pre-Android; the mid-range Android (TR locale) smoke re-enters as a release gate in the Android enablement follow-on (M6.5).**

### Golden harness (M1.4)
Plain `flutter_test` `matchesGoldenFile` + a small in-repo harness (`app/test/support/golden/`), zero added deps. eBay's `golden_toolkit` is discontinued; Betterment's `alchemist` was evaluated (2026-07) and is maintained, but its cross-OS block-text mode defeats the real-font rendering we want and its gallery ergonomics don't pay for a dependency at our screen count. Real bundled brand fonts (Rubik + Noto Sans / Noto Sans Arabic) are loaded once in `app/test/flutter_test_config.dart`, so goldens exercise real Arabic shaping and the 1.5/1.7 line-heights ‚Äî never Ahem. Goldens are **Linux-canonical**: rendered on the Linux dev box, verified in the ubuntu `quality` job. macOS renders text differently (antialiasing/hinting), so macOS machines must **never** run `flutter test --update-goldens` ‚Äî re-baseline happens on Linux or via CI. Comparison is exact (no tolerance comparator), which makes goldens engine-version-sensitive: expect intentional churn on a Flutter upgrade, re-baselined behind the explicit W4 golden-update flag. Mismatch diffs land in `failures/` next to the test (gitignored) and upload as CI artifacts.

## 2. CI validation

`ci.yml` (every push/PR): format ‚Üí analyze (RTL lint included) ‚Üí unit + widget (incl. goldens) ‚Üí coverage gate ‚Üí debug build; plus the ubuntu `functions-rules` job (Functions + rules emulator suites with the TS coverage gate, M2.1). **App-side seam decision (M2.2, per resume-prompt):** widget/unit tests fake the pairing repository and share-launcher seams ‚Äî `cloud_functions`, `share_plus`, and `app_links` all drive platform channels that throw in the plain `flutter test` VM ‚Äî so the real `createInvite` wire path is exercised by `integration_test/pairing_emulator_test.dart` (the main-only `integration-emulator` job, which since M2.2 also builds `functions/lib` and boots the functions emulator with `USE_FUNCTIONS_EMULATOR=true`), and the Function itself is proven per-PR in the functions-rules suite. Custom-scheme *delivery* (OS ‚Üí `app_links`) is unexercised under `flutter test`; it's covered by the M2.3 on-device deep-link test. `release.yml` (tags): emulator integration suite ‚Üí E2E on the iOS simulator matrix (current-1/current) ‚Äî iOS-first (ADR-006): the Android API 30/34 emulator matrix re-enters this release gate in the Android enablement follow-on (M6.5) ‚Üí signed builds ‚Üí distribute. Branch protection: no green, no merge (rule #7).

## 3. Coverage goals

| Scope | Target | Gate |
|---|---|---|
| `domain/` (all features) | 90% | Hard fail <85% |
| Functions (TS) | 85% | Hard fail <80% |
| Overall Dart | 70% by M6 | Ratchet: starts 60%, +2%/milestone, never lowered |
| Screens with goldens | 100% of P0 screens | PR checklist |

Coverage is a floor, not a goal ‚Äî acceptance criteria in `implementation-plan.md` define sufficiency.

## 4. Policies

Flaky test ‚Üí quarantined same day with an issue; >2 quarantined = next session objective is stabilization. Bug fix ‚Üí regression test in the same commit, named after the issue. Test speed budget: unit+widget suite <90s locally ‚Äî protect it (fakes over emulators at unit level). LLM-dependent tests never call live models in CI (fixtures only); safety-path tests are the exception and run against the proxy's filter layer, which is deterministic. No test reads real user data, ever.
