# Test Suite Strategy — Hayati

Strict TDD (rule #6). The pyramid, tooling, and gates below are enforced by CI, not by good intentions.

## 1. Layers

### Unit tests (`test/`, mirrors `lib/`) — the wide base
Pure-Dart domain layer makes these cheap and fast. Mandatory targets: streak engine (property-based: gaps, grace tokens, timezone edges, DST, Ramadan quiet-window interactions) — **landed M3.4 server-side** (the engine lives in `functions/src/streak/`, TS + fast-check, because the answerReveal trigger is its only writer per ADR-012; the app carries just a display-only zero-state-parity read model, so no Dart engine twin exists by design; Ramadan quiet-window interactions stay deferred with seasonal content, ADR-011); pairing state machine (races, expiry, re-pair attempts); question selection (register/seasonal/no-repeat determinism); entitlement mapping (one purchase → two partners; expiry → both) — **landed M4.1 server-side** (the decision core lives in `functions/src/entitlements/`, TS + fast-check, because the `revenueCatWebhook` Function is the mirror's only writer per ADR-013; the app carries just the summary read model + the `isPremium` derivation, provider-tested). **M4.3 (ADR-015) added `TRANSFER`**: the pure `gateTransfer`/`planTransfer`/`revokedLane`/`revokeLane` core is table-tested over the full case matrix (within-couple = hold; cross-couple = revoke the loser only; unknown/anon/empty destination = hold; oversized = hold at the pre-read gate; multi-target = one tombstone per losing couple) and the **convergence property now folds a TWO-COUPLE world with transfers mixed into the event multiset** — it is mutation-verified: making the tombstone copy the loser's prior lane facts, entitling a gainer, or dropping the tombstone-when-no-lane-exists rule each break order-independence and turn it red. The emulator suite proves the hazards end-to-end (the shared-Apple-ID within-couple restore leaves the doc BYTE-unchanged; the reinstall's anonymous destination never downgrades a payer; a transfer arriving BEFORE the purchase it moves cannot be resurrected by the late purchase; duplicate/stale transfers write nothing), and **one e2e wire test POSTs RC's real transfer body (no `app_user_id`) — the regression that the M4.1 envelope contract would have 400'd**; pack validator; crisis-lexicon matcher (TR/AR/EN seed lists) — **landed M5.1 server-side** (ADR-016: the pure normalizer + detector live in `functions/src/coach/`, TS + fast-check, because `coachProxy` is the only consumer; evasion resistance is normalization-based with **one property-mutation class per fold step** — NFKC forms, Cf/VS injection, tashkeel/tatweel, alef/ى/ة/hamza folds, Arabic-Indic digits, TR İ/ı + TR diacritic-strip, homoglyphs, repetition inflation, separator injection incl. token-mode entries, leet, cross-message splitting — so removing ANY fold turns CI red; seeds + canonical evasion variants are byte-pinned in the policy-headed `functions/test/fixtures/coach-crisis-seeds.json` with per-language spread guards). Tools: `flutter_test`, `mocktail`, fakes over mocks for repositories, builder-pattern test data (`aCouple().paired().withStreak(12).build()`).

**Pack validator — live since M3.1** (`content/validator/`): a pure core (`validator_core.dart`, no IO — schema fields/patterns/enums/depth bounds, cross-pack id uniqueness, `packId`↔filename↔`locale` consistency, reviewedBy warning tier per ADR-007) behind a thin IO shell (`validate.dart`, exposing an in-process `runValidator` and the ADR-010 `--sync` mode). **Runner decision:** `content/` has no pubspec, so `package:test` doesn't apply; the tests are a plain-`dart` self-checking script (`validate_test.dart`, `tool/`-script mold — 63 checks over every violation class plus shell round-trips against a temp root; any failure exits non-zero). The shell also cross-checks the validator's vocabulary against `question-pack.schema.json` itself on every run, so schema edits that outpace the core fail CI loudly.

The suite boots the FULL app (`HayatiApp` + the four new seam overrides + a signed-in fake auth and a paired profile, so real couple content exists to hide) and pins: cold start locks and the correct PIN reveals *that same content*; background-return past the 60s grace re-locks and within it does not; `.inactive` alone raises the shield but never starts the grace clock (the share sheet and the biometric prompt both pass through it); a cold-start deep link while locked is *captured* into `pendingInviteProvider` yet neither findable nor tappable until unlock, then renders; a pushed route survives a re-lock on the Navigator stack; and — the free-tier probe — with no lock record there is no lock UI anywhere and the homes behave as before. Attempt-bounding, the S019-class generation-guard race (a wrong-attempt persist in flight when the sign-out wipe fires must NOT resurrect the record), the counter-refund race (a biometric revoke must not roll back a consumed attempt), the degraded-boot resume reconcile, and the enrollment-change auto-revocation are pinned at the controller level against a `FakePinLockStore` with a `writeGate` and a `FakeBiometricAuthenticator` with a `probeGate` — the gates are what make the races *provokable* rather than hoped-about, and each guard was **mutation-checked** (neutering it turns exactly its own tests red). Guarantees no fake can carry are pinned by **source-sentinel tests** (the coach ARB digit-run precedent): `biometricOnly: true` in the local_auth adapter (no fake has a device passcode to fall back to, and plain `deviceOwnerAuthentication` would accept the *device passcode* — an app-PIN side door); the total try/catch around every local_auth call (3.x *throws* where the seam promises a bool); and — because `flutter test` touches no platform channel while CI compiles the app but never RUNS it — **the `hayati/device_privacy` channel's name, all four method names, and the discreet icon's name across the Dart constant, the `.appiconset` directory and all three pbxproj configs.** That last one guards a failure mode every other check calls green: a drifted name compiles perfectly and ships a feature that is simply *absent* (the M6.1 review's IOS-1 class). Each of these, and every concurrency guard in the lock controller, is **mutation-checked** — neuter it, confirm exactly its own tests turn red, restore; a first-draft race test in M6.1 was found to pass against the very bug it was written for, and only the mutation check caught it. The three device-only adapters (Keychain, local_auth, the icon channel) are never imported by a test — `flutter test` touches no platform channel, and they stay out of the coverage denominator. Goldens are the RTL regression net — an un-mirrored arrow fails the build, not the Riyadh user. (The lock keypad is pinned `TextDirection.ltr` on purpose: numeric pads are not mirrored in RTL on any platform, and the AR goldens pin 1-2-3 order.) Golden updates require an explicit flag in the PR (see `agent-workflows.md` W4).

### Functions + rules tests (`functions/test/`, M2.1) — vitest against the Emulator Suite
**Tooling decision (Session 007): vitest** over jest — TS-native via esbuild, first-class v8 coverage with `thresholds` as the §3 hard gate (proven to exit non-zero below 80%), one config. `fast-check` for property tests (code charset/length/collision). Because esbuild strips types **without checking them** and the build tsconfig covers only `src/`, `npm run typecheck` (`tsconfig.test.json`, bundler module semantics) is the step that typechecks `test/` — wired into the CI job. ### Golden harness (M1.4)
Plain `flutter_test` `matchesGoldenFile` + a small in-repo harness (`app/test/support/golden/`), zero added deps. eBay's `golden_toolkit` is discontinued; Betterment's `alchemist` was evaluated (2026-07) and is maintained, but its cross-OS block-text mode defeats the real-font rendering we want and its gallery ergonomics don't pay for a dependency at our screen count. Real bundled brand fonts (Rubik + Noto Sans / Noto Sans Arabic) are loaded once in `app/test/flutter_test_config.dart`, so goldens exercise real Arabic shaping and the 1.5/1.7 line-heights — never Ahem. Goldens are **Linux-canonical**: rendered on the Linux dev box, verified in the ubuntu `quality` job. macOS renders text differently (antialiasing/hinting), so macOS machines must **never** run `flutter test --update-goldens` — re-baseline happens on Linux or via CI. Comparison is exact (no tolerance comparator), which makes goldens engine-version-sensitive: expect intentional churn on a Flutter upgrade, re-baselined behind the explicit W4 golden-update flag. Mismatch diffs land in `failures/` next to the test (gitignored) and upload as CI artifacts.

## 2. CI validation

## 4. Policies

Flaky test → quarantined same day with an issue; >2 quarantined = next session objective is stabilization. Bug fix → regression test in the same commit, named after the issue. Test speed budget: unit+widget suite <90s locally — protect it (fakes over emulators at unit level). LLM-dependent tests never call live models in CI (fixtures only); safety-path tests are the exception and run against the proxy's filter layer, which is deterministic. No test reads real user data, ever.
