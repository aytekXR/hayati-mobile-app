## Objective — Session 012: M3.2 — daily rollover Function: timezone buckets, deterministic question selection, `couples/{cid}/days/{yyyymmdd}` + rules

Second slice of M3 (`implementation-plan.md`). M3.1 closed in Session 011: the content pipeline is enforced reality (validator + CI + `content/packs/` authoring home with byte-sync, ADR-010) and the app serves packs through the generalized `QuestionPack`/`Question` model (register carried, `seasonalWindow` surfaced). This session builds the server half of the daily loop — the scheduled rollover that assigns each couple its day's question:

1. **`question_rollover` scheduled Function (TS, europe-west1):** per timezone bucket (couples grouped by their `timezone` field), assign the day's question for each couple by writing `couples/{cid}/days/{yyyymmdd}` (dayKey in the couple's LOCAL calendar date, same shape as `soloAnswers`). Idempotent: re-runs and overlapping buckets must never double-write or reassign a different question to an existing day doc.
2. **Deterministic selection (pure TS core, unit-tested):** next unseen question from the couple's pack honoring register + seasonal rules — no repeats against the couple's `days` history; seasonal questions only inside their window (decide + document the window→date mapping for `ramadan`/`eid`/`new_year`, or explicitly defer non-evergreen selection with a documented evergreen-only policy); deterministic for a given (couple, date, pack version) so the client can prefetch offline (`architecture.md` §4). Selection inputs come from the couple's `packConfig` (shape decided this session — minimum: packId; document in architecture §3).
3. **Pack source for Functions (decide + document, likely ADR-011):** how the Function reads validated pack content — bundle `content/packs/` into the functions build (a copy step in `npm run build`) vs Firestore-hosted packs. Personal-use scale favors the build-time bundle; whatever is chosen, the validator remains the single content gate.
4. **Rules:** `couples/{cid}/days/{yyyymmdd}` member-only read, **client writes denied entirely** (function-only via Admin SDK), mutation-tested in the weakened-rules harness. This is the doc the M3.3 reveal invariant ("partner answer unreadable pre-answer") will extend — keep the day doc's field surface minimal (`questionId`, `assignedAt`, pack version) and documented.
5. **Timezone-bucket scheduling:** decide the bucket granularity (per-timezone scheduled runs vs a single hourly run that computes which timezones just crossed local midnight — the M2.3 timezone allow-list is the input domain) and prove DST correctness with unit tests (a couple in Europe/Istanbul vs Asia/Riyadh gets its new day at ITS local midnight, incl. DST-shifting zones).
6. **Docs-with-code:** `architecture.md` §3 (couples/days/packConfig shapes) + §4 (rollover flow status), `test-suite.md` §1/§4 (functions layers), `implementation-plan.md` (M3 progress line), ADR-011 if the pack-source decision is genuinely weighed, this file.
